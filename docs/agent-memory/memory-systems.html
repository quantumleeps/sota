03 · Memory Systems

Building the store behind the window

The persistent layer: systems that decide what to write down, how to maintain it, and what to pull back. The clearest trend across the corpus is a shift in who decides — from hand-coded heuristics toward learned, query-aware control of memory itself.

A common spine: write → manage → read

The memory survey gives the organizing loop every system here instantiates: a read operator and an update operator that "summarizes, deduplicates, scores, resolves contradictions, deletes — not a simple append operation" Du '26 Definition. Systems differ mainly in their control policy — heuristic, prompted, or learned.

The production baseline: Mem0

Mem0 is the reference for "external memory beats stuffing the window." It extracts dialogue facts, lets the LLM choose among four operations, and retrieves selectively; a graph variant (Mem0ᵍ) adds relational structure Chhikara+ '25 (Mem0) Proposal. On the LOCOMO benchmark it lands close to full-context accuracy (LLM-judge J ≈ 66.9%, vs full-context 72.9%) while cutting p95 latency ~91% and token cost >90% Empirical. The honest caveat is baked into the same table: full-context still edges it on raw accuracy — memory buys efficiency, not a free accuracy win.

Compaction as memory: ReadAgent's gist

ReadAgent shows that a memory store can be built purely by prompting. It paginates long text at LLM-chosen "pause points," compresses each episode into a fuzzy gist, keeps the gist set in context as a navigable index, and re-expands verbatim pages only when a task needs them Lee+ '24 (ReadAgent) Definition. A small but telling craft detail:

We use the word "shorten" in the prompt … Using the word "summarize" tended to produce a restructured summary in our experiments. — ReadAgent Lee+ '24 (ReadAgent), §Memory Gisting

It extends effective context 3.5–20× while beating retrieval and full-context baselines (see Evidence) Empirical — the two-tier "fuzzy index + verbatim look-up" pattern that recurs throughout the hierarchy work.

The pivot: learned memory management

Three 2026 systems make the management policy itself learned. AgeMem folds long- and short-term memory operations into the agent's own policy as tool actions and trains them end-to-end, broadcasting the final task reward to every intermediate memory decision via step-wise GRPO Yu+ '26 (AgeMem) Proposal. RL doesn't just tune — it changes behavior, shifting retrieval to be "more selective and query-driven" and adding +8.5 points over a no-RL ablation Empirical.

MemSkill goes further and makes the operations learnable and evolvable — a shared "skill bank" that a designer grows from hard cases, so the agent discovers domain-specific memory tactics rather than inheriting fixed add/update/delete primitives Zhang+ '26 (MemSkill) Proposal:

the evolved skill bank reflects recurring information needs from the data, rather than a fixed notion of what to remember. — MemSkill Zhang+ '26 (MemSkill), §Case study

Its span-level (not turn-level) extraction is also the efficiency story: comparable or better quality at 215 LLM calls where a strong baseline used 1,288 Empirical.

BudgetMem attacks a different waste: building memory offline, query-agnostic, "build once, use always." Instead it defers computation until a query arrives and routes each pipeline module to a Low/Mid/High tier, learning when to spend compute Zhang+ '26 (BudgetMem) Proposal — yielding a controllable accuracy–cost frontier and tunable latency (3,881 ms → 1,167 ms as the cost weight rises) Empirical.

▦ A real result worth holding onto

Across AgeMem, MemSkill, and BudgetMem, the same lever pays off: making memory management learned and query-aware beats fixed schedules on both quality and cost on long-horizon benchmarks Yu+ '26 (AgeMem) Zhang+ '26 (MemSkill) Zhang+ '26 (BudgetMem). Note the shared limitation, though: all three are single-team evaluations on Qwen/LLaMA backbones with no significance testing.

⟡ My inference — not from the papers

The arc is unmistakable: heuristic (Mem0) → prompted (ReadAgent) → learned (AgeMem, MemSkill, BudgetMem), with control migrating into the policy. But the benchmark page is where I'd temper the enthusiasm: MemoryArena finds that bolting an external store onto an agent gives "no consistent gain" over raw long context. My read is that how memory is managed is advancing faster than whether the agent can use it — which is why the co-design lesson on the Synthesis page matters more than any single architecture here.

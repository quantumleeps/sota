03 · Keeping Context Inside the Budget

The context wall: managing what accumulates

A long-horizon coding task floods the window with file contents, logs, and command output. Three papers attack the resulting "context wall" from different angles — pruning observations, caching orientation, compressing into latent tokens — and together they deliver the corpus's clearest design rule: preserve the decision-relevant tokens exactly; the compression ratio is the wrong objective.

Reading accumulates faster than anything else

The pressure comes overwhelmingly from one source. SWE-Pruner '26 measures that read-type operations consume 76.1% of a coding agent's total tokens (Claude Sonnet 4.5 on SWE-bench Verified), versus 12.1% for execution and 11.8% for editing — a pattern that holds for GLM-4.6 at 67.5% Empirical. The raw retrieved context, "often thousands of lines," is what walls the agent in.

Negative result: implicit compression breaks multi-step coding

The tempting fix is to compress past context into a denser representation. Gelvan+ '26 tests exactly this — an In-Context Autoencoder that encodes context into continuous "memory tokens" — and reports a clean negative result Empirical:

While the method performs well on single-shot common-knowledge and code-understanding tasks, our experiments demonstrate that it fails on multi-step agentic coding tasks. — Implicit Context Compression Gelvan+ '26, Abstract

The diagnosis is mechanistic and directly relevant to coding: small reconstruction errors (a hallucinated file path, swe-agent.com/latest/ → swe-agent.com/agent/latest/) are harmless in one-shot QA but compound across steps, and the exact-match str_replace_editor tool then fails on the corrupted observation Theory. Tellingly, the effective compression rate did not correlate with downstream performance at all — squeezing harder neither helped nor explained the failure Gelvan+ '26.

And so does naive history persistence

The opposite reflex — just keep everything — fares no better. Peek '26 shows that persisting full interaction history ("Shared Chat") inflates input tokens roughly 9–10× (e.g. 2.85M → 28.76M on one task) for negligible or negative quality, and on a coding-style context-learning benchmark the solve rate actually drops 14.0% → 12.0% Empirical. More raw context is not more usable context.

What works: selective, structure-preserving reduction at the right boundary

The two positive systems both reduce context, but they reduce it selectively and keep what survives intact.

Prune the observation, not the trajectory Empirical

SWE-Pruner inserts a 0.6B "neural skimmer" as middleware at the agent–environment boundary: the agent emits a natural-language Goal Hint ("How is authentication handled?"), and the pruner filters each file-read to the relevant lines before it reaches the agent SWE-Pruner '26. Because it prunes at line rather than token granularity, it preserves syntax — AST correctness 87.3% vs 0.29% for a token-level compressor (LLMLingua-2):

In one case the pruner converted a resource-exhaustion failure into a success: a Django task that the baseline abandoned after 164 steps / 7.0M tokens was solved in 56 steps / 1.17M tokens — an 83.3% token reduction Empirical. The authors frame pruning as "not merely an engineering optimization but a cognitive strategy that enables more efficient problem-solving behaviors."

Cache orientation, not history Empirical

Peek maintains a small, constant-sized context map inside the prompt that caches reusable orientation knowledge about a recurring context — what it contains, how it is organized, which schemas and constants recur — built and maintained by a Distiller→Cartographer→Evictor policy with per-item IDs and a hard token budget Definition Peek '26. Evaluated on OpenAI Codex among other agents, it improves quality 6–34% over the base agent and beats the prompt-learning method ACE by 7.8–15% while using 93–145 fewer iterations and costing up to 5.8× less:

a context map is an agent-level artifact whose value comes from curated meaning. — Peek Peek '26, Introduction

An ablation is the key result for this page: across a 4× budget range, the presence of a map mattered more than its size — and separating extraction (Distiller) from editing (Cartographer) was essential, because without it "non-contextual facts leak into the map and updates become noisy" Empirical Peek '26.

⟡ My inference — the unifying rule across these three papers

Read together, the three results point away from "compress the context" and toward "protect the load-bearing tokens, discard the rest, and never lossily re-encode what a tool will match exactly." The evidence: compression ratio doesn't correlate with performance Gelvan+ '26; map presence beats map size Peek '26; line-level pruning preserves syntax where token-level shreds it SWE-Pruner '26; and on the previous page, missing core context hurt more than redundant context SWE-Explore '26. The shared objective is fidelity of the relevant subset, not bytes saved. A coding agent is a brittle exact-match machine downstream of its context; this corpus keeps rediscovering that. This synthesis is mine, not any single paper's.

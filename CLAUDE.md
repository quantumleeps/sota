# SotA — state-of-the-art review hub

A self-contained, **topic-agnostic** working repo that turns a research topic into a hyperlinked, provenance-cited review and publishes it as one topic in a shared hub on GitHub Pages. `cd` here and run Claude Code. Three bundled skills compose into one pipeline: **`/arxiv-paper-finder`** discovers and ranks candidate papers, **`/sota-site`** turns a chosen set into a published topic, and **`/sota-audio`** (optional follow-on) gives that topic an audio-native "Listen" page narrated for the ear.

**Live:** https://quantumleeps.github.io/sota/

## Workflow: discover → review → publish

1. **Discover — `/arxiv-paper-finder`.** Give it a topic ("explore LLM agent memory"). It pulls a candidate superset from arXiv / OpenAlex / Hugging Face (Semantic Scholar too, if `S2_API_KEY` is set), ranks by relevance (local embeddings) + recency + citations + has-code + social, and returns a shortlist. First use: `pip install -r .claude/skills/arxiv-paper-finder/requirements.txt`.
2. **Review.** Read the ranked list with the user and curate it; re-rank with different weights (cheap — no re-fetch) until the set is right.
3. **Hand off.** Re-run the ranker with `--emit-list <slug>.papers.txt` to write a sota-site-ready list (`<arxiv-id>  # <title>` per line).
4. **Publish — `/sota-site`.** Feed that list as the corpus for a new topic: fetch full sources → read via parallel agents → digest → synthesize `docs/<slug>/` → regenerate the hub → verify → publish.
5. **Narrate (optional) — `/sota-audio`.** Run right after `/sota-site` in the **same session** (while the digests/synthesis are still in context): it authors one continuous, audio-native narration of the topic, publishes a `docs/<slug>/listen.html` page (player + transcript), and synthesizes the spoken MP3 on demand via Fish Audio / ElevenLabs. Authoring the script needs no API key; audio is opt-in. First synth use: `pip install -r .claude/skills/sota-audio/requirements.txt`.

**Slug / versioning:** if the topic already exists (see *Current topics*), use a `-v2` slug — e.g. exploring agent memory again → **`agent-memory-v2`**. Each version is a fresh corpus read in isolation and added alongside the existing topic on the hub (the skills never merge corpora unless told to).

## Layout

```
.claude/skills/sota-site/   the skill (canonical copy; travels with this repo)
CLAUDE.md  README.md  .gitignore
docs/                       ← PUBLISHED to GitHub Pages (source: main /docs)
├── index.html              the hub landing (topic cards + nav)
├── styles.css  .nojekyll   ONE shared stylesheet at the docs root
└── <slug>/*.html           one topic per dir; each has its own corpus.html
topics-corpora/             ← WORKING material. GITIGNORED. never published.
└── <slug>/
    ├── corpus/ {papers/<id>/…, papers/index.json, papers.txt, meta.json}
    └── digest/
```

- **Published** (committed): `docs/`, `.claude/`, `CLAUDE.md`, `README.md`, dotfiles.
- **Working** (gitignored): everything under `topics-corpora/` — corpora and digests.
- Topic pages link `../styles.css` (shared) and `../index.html` (the "↑ All SotA" up-link). GitHub Pages serves the `docs/` folder, so the skill and working files ride along in the repo without being exposed.

## Adding a topic

Hand the skill a paper list (arXiv IDs/URLs, optional lab writeups) and a kebab slug. The skill (`.claude/skills/sota-site/SKILL.md`) runs: fetch → read every paper in full via parallel agents → digest → synthesize `docs/<slug>/` → regenerate the hub → verify → publish. Two options worth knowing:

- **Review vs deep dive** (default: review). A *review* reads exactly the given set. A *deep dive* (opt-in) also harvests the next level of cited references from an anchor paper; it is clearly labelled and reports its reference count. Set `meta.json`'s `"mode"` and (for a deep dive) its `"anchor"`, then `scripts/fetch-paper-refs.sh`.
- **Date cutoff** (default: none). Export `SOTA_AFTER` / `SOTA_BEFORE` (YYYY or YYYY-MM) before fetching to programmatically drop papers outside the window (from each arXiv id's YYMM — applies to harvested references too). Preview with `scripts/arxiv-date.py --after 2026 --report < papers.txt`.

## Corpus manifest

Each topic's `corpus/papers/index.json` is the manifest of record (generated by `scripts/build-index.py` from the filesystem + curated `corpus/meta.json`). It carries each source's exact `read` entrypoint, role, format, the `mode` (review/deep-dive), and — for a deep dive — the anchor's references triaged by title. `corpus/papers.txt` is the human input feed. **Read `index.json` first when ingesting — do not glob.**

## Publishing

`scripts/publish_pages.sh <repo-root> [repo-name]` from this directory: commits + pushes, creates the repo on first run, and ensures Pages serves `main` `/docs`. Publishing is public and effectively permanent — confirm before running. Content derives from public papers/lab writeups; it is not re-hosted.

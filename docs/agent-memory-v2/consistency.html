Page 03 · the validity problem

Keeping memory true

The most consistent — and most alarming — finding in this corpus is that getting information into memory is largely solved, but keeping that memory correct, current, and authoritative is not. Agents store the right fact, retrieve the right fact, and then answer from the stale one. This page collects the corpus's evidence on conflict, staleness, updates, and the discipline of maintaining state instead of hoarding it.

▦ The shared failure

Across the validity-focused papers the anti-pattern is the same: append-only accumulation. Memory grows, old entries are never retired or overwritten, and at answer time the agent cannot tell which version is current. The fixes diverge — write-time adjudication, version-aware patches, retention-aware updates — but the diagnosis is shared.

Implicit conflict and the "adjudication gap"

STALE isolates a failure mode prior benchmarks miss: implicit conflict, where a later observation invalidates an earlier memory without any explicit negation, so detecting it needs inference rather than string-matching. STALE '26 Definition It probes three behaviours — can the model resolve the current state, resist a question that presupposes the stale state, and act on the update in a downstream task — across 400 expert-validated scenarios.

The results are blunt. The best model evaluated reaches only 55.2% overall; most external memory frameworks score below 10%, and several do worse than the plain model. STALE '26 Empirical A diagnostic on one framework shows the new evidence is present in retrieval for ~77% of cases — visibility is not the problem; authority is.

updated evidence can be stored and retrieved, but it does not reliably become the basis that governs subsequent answers. We term this the current-state adjudication gap. — STALE STALE '26, §How Do Memory Frameworks Adjudicate Current State

The proposed fix is to move adjudication to write time: a prototype (CUPMem) that marks each memory KEEP / STALE / REPLACE / UNKNOWN as it ingests, and searches for propagated invalidations, lifts a weak backbone from 8.7% to 68.0% — though its own downstream-action score stays lower, so the gap is narrowed, not closed. STALE '26 Empirical

Models can identify outdated information under explicit probing, yet still comply when a query presupposes the outdated state. — STALE STALE '26, §Overall Performance

State collapse and version-aware updates

EvoArena diagnoses the structural reason memory goes stale: agents keep memory as "a single latest state," which collapses when different versions of an environment need different behaviour. EvoArena '26 Theory Its EvoMem patch history (see Architectures) treats a non-additive update — one that revises or overwrites prior memory — as a first-class, traceable event with pre/post state, rationale, and evidence, rather than silently overwriting. EvoArena '26 Proposal Empirically it preserves more complete evidence sets (row-level capture rises most on temporal-trajectory questions) and cuts code-regression rates — but the authors are candid that on the hardest preference-conflict cases, retrieving historical patches can add ambiguity about which state is current. EvoArena '26 Empirical

Maintenance over accumulation

WorldMemArena turns the validity question into an evaluation: its four-stage lifecycle (write → maintain → retrieve → use) lets it localise where memory breaks, and the "maintain" stage is where most systems fail — they are weak at updating and at rejecting distractors, accumulating rather than maintaining a consistent state. WorldMemArena '26 Empirical It scores an update as successful only when the revised memory is kept and the old version is removed or overwritten — a strict definition most systems don't meet. WorldMemArena '26 Definition Two further findings reinforce the theme: higher storage quality does not translate into better answers, and increasing retrieval scope can hurt by surfacing redundant or outdated evidence. WorldMemArena '26 Empirical

Memory requires consistent state maintenance, not continuous accumulation. — WorldMemArena WorldMemArena '26, §Discussion

Consolidation, forgetting, and what not to store

If accumulation is the disease, deliberate forgetting is part of the cure — and several systems build it in. FluxMem prunes interfering links and consolidates recurrent trajectories offline, with a maturity metric (PEMS) that signals when consolidation should stop rather than running forever. FluxMem '26 Proposal Bayesian-Agent makes forgetting an explicit, auditable decision — a skill whose posterior success falls is retired, not left to rot. Bayesian-Agent '26 Proposal And Personalize-then-Store attacks the upstream decision entirely: the cleanest way to avoid stale clutter is to never store transient sessions in the first place — though it shows current gating can't yet decide this reliably. PerMem '26 Empirical

⟡ My reading — the binding constraint moved

Read together, these papers relocate the hard problem. A few years of memory work optimised recall; this corpus says recall is no longer where systems fail. The bottleneck is adjudication — deciding, among everything stored and retrieved, which belief currently governs the answer. STALE names it directly, WorldMemArena measures it as failed "maintain," and EvoMem's whole design is an attempt to make the current state recoverable from a history of changes. If that reading is right, the next round of memory systems will be judged less on what they can retrieve and more on whether they can say "this fact is now false."

⚠ Why this matters beyond benchmarks

A memory that confidently serves stale state is a safety problem, not just an accuracy one: STALE shows agents comply with questions that presuppose an outdated fact, and EvoArena notes persistent adaptive memory "can surface outdated or context-specific information if the agent fails to judge whether a prior state remains applicable" EvoArena '26 STALE '26. In a deployed agent — booking, code, health, finance — acting on a superseded memory is a wrong action, not a wrong answer.

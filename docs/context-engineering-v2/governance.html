06 · Governance & Provenance

Control at the boundary where agents act and contribute

As agents move from suggesting code to executing actions and opening pull requests, control shifts to two boundaries: where a proposed action becomes a real one, and where a machine's contribution enters a human project. This corpus puts deterministic governance and machine-readable provenance at both — though its hardest numbers come from outside the coding domain.

The execution boundary: govern the action before it runs

Shi+ '26 names the execution-boundary problem — proposed actions must be governed before they are executed — and proposes a model-agnostic Organizational Control Layer that intercepts each agent-generated action and either approves, revises, blocks, or escalates it against policy, emitting an audit trace Definition:

deployment-grade agent systems should separate proposal generation from environment-facing execution. — Organizational Control Layer Shi+ '26, Abstract
12% → 96%
Valid (policy-compliant) success rate without vs with the control layer — baseline "success" was largely non-compliant. Shi+ '26
88% → 0%
Unsafe-execution rate; executed policy violations fell from 205 to 0 across 50 episodes. Shi+ '26
e-commerce
The domain is adversarial buyer–seller negotiation, not coding — read these as transferable mechanism, not a coding result. Shi+ '26

The transferable parts for coding agents are the four-outcome vocabulary {Approve, Revise, Block, Escalate}, the observable-vs-hidden constraint split (an agent acts with incomplete repo/policy context, so escalation handles what it cannot see), and the audit trace recording why each action was approved or blocked Proposal Shi+ '26. The paper's own caveat: the layer over-constrains the agent in cooperative thin-margin cases, a genuine safety–utility trade.

▦ The execution boundary is already in production harnesses

This is not only a proposal. The Claude Code analysis documents a deny-first permission system with seven modes as the deterministic enforcement layer, deliberately split from the probabilistic guidance of CLAUDE.md — "a compromised or adversarially manipulated model cannot override the sandboxing, permission checks, or deny-first rules" Liu+ '26 Definition. And the harness survey lists multi-tier permissions and human-in-the-loop gates as durable harness state, naming "governed" a target property and "executable accountability" the goal Ning+ '26 Proposal.

The contribution boundary: machine contributors meet human projects

The second boundary is social. Manita & Amari '26 compare six open-source projects' AI-contribution policies and identify the asymmetry that breaks existing governance Empirical:

AI lowers the cost of producing contributions but does not lower the cost of trustworthy human review. — Regulating the Machine Contributor Manita & Amari '26, §1

Their six-dimensional taxonomy (disclosure, responsibility, oversight, licensing, enforcement, maintainer workload) yields Policy Maturity Scores from LLVM 20 down to the Linux Foundation 7, and surfaces three findings relevant to context engineering Empirical:

  • Provenance trailers are codified-context artifacts. The Generated-By: / Assisted-By: commit-message labels are machine-parsable provenance — the same class of durable, version-controlled artifact as repo instruction files, here used for accountability.
  • Autonomous agents fall through policy gaps. Only 2 of 6 projects explicitly address the autonomous-agent case; the rest assign responsibility to "the contributor" without addressing the case where no human contributor exists.
  • Maintainer review capacity is the universally ungoverned dimension. No policy and no reviewed regulatory framework provides rate limits, cooldowns, or triage to protect reviewers — documented against real incidents (a bot's PRs closed under matplotlib's policy; an adversarial agent on SymPy).

Provenance as both context and control

Several artifacts from earlier pages are, in this light, governance instruments. PROJECTMEM's immutable plain-text log "doubles as an automatic provenance/reproducibility trail," with its pre-action judgment gate enforcing the project's own history — Memory-as-Governance PROJECTMEM '26. SkillWiki's Provenance Graph and reversible Git-style reviews keep skill evolution auditable, with users overriding autonomous governance SkillWiki '26. And the self-optimizing harness keeps full audit logs and requires human approval for sensitive edits, precisely because "harness updates distilled from compromised trajectories could entrench bad behavior" Pan+ '26 Proposal.

⟡ My inference — not from the papers

Governance is the property the rest of this corpus implies but rarely measures. The recurring instinct — keep enforcement deterministic and outside the model, at the boundary — is the same one Liu+ found already shipping in Claude Code (probabilistic guidance vs deterministic permissions) Liu+ '26, that Shi+ formalize as a control layer Shi+ '26, and that Ning+ name a target property Ning+ '26. My reading is that provenance and governance are converging on the same artifacts: a log/trace/graph that is simultaneously the agent's context, its audit trail, and its policy. But I'd hold the claim loosely — the only large governance effect sizes in this corpus come from a non-coding domain Shi+ '26, and the one coding-adjacent governance study concludes the core problem (trustworthy review at machine speed) is unsolved Manita & Amari '26. This is the corpus's most-asserted, least-measured dimension for coding specifically.

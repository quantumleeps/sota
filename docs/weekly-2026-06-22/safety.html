02 · Safety & Robustness

The agent as an attack surface

Three papers in this sample move "agent safety" off the model and onto the substrate the model runs in. One audits the agent's runtime as privileged software; one shows the agent's own tool choices leak privilege in ways safety-alignment never touched; one builds robustness into the design search instead of patching it on. Read together they make a single argument: the dangerous part of an agent is increasingly the code and the choices around the model, not the refusal behaviour of the model itself.

The runtime is the unaudited boundary

Zhang+ '26 reframes a local LLM agent as "not merely a model wrapper… a privileged software runtime": a pipeline of prompt builder, parser, tool dispatcher, skill loader, memory writer, network client, and permission gate, all running at the user's privilege with no internal boundary between components. Prior agent-security work targets prompt injection, malicious skills, and black-box behaviour; the implementation layer that actually executes goals had not been systematically audited.

even an agent that correctly refuses every dangerous prompt can still be exploited through its runtime — ClawAudit Zhang+ '26, §Implications

▦ Domain-specific rules find what generic scanners miss Empirical

On a held-out set of 217 real source-level advisories, generic scanners detect a minority — Semgrep Pro 21.7%, CodeQL security-extended 13.8%. Adding ClawAudit's STRIDE-derived, agent-runtime rules lifts recall to 66.8% (Semgrep) and 75.1% (CodeQL) — and the gain generalizes temporally (train→test recall gaps within ±4 pp), so it isn't memorization Zhang+ '26. Empirical

The paper is unusually honest about where rule-based auditing stops. On the live codebase, precision is low — a stratified manual audit found ~12% true positives (6 of 50 sampled findings) — and rule quality is bimodal: all six true positives came from six distinct narrow-scope rules, while high-volume broad rules produced none Zhang+ '26. And the residual gap is principled: vulnerability classes like incomplete denylists (CWE-184) and information exposure (CWE-200) have no syntactic surface for a rule to intercept.

the boundary is therefore not where static analysis stops working in the abstract, but where the bug stops having a syntactic surface to intercept — ClawAudit Zhang+ '26, §RQ5

⚠ A scaling problem hiding in the tooling

Canonical taint queries don't scale to agent codebases: 21 of 104 CodeQL security-extended queries failed to terminate within a 4-hour-per-query budget on the 16,433-file runtime — including command-injection and request-forgery queries, exactly the classes one most wants to run Zhang+ '26. The safety tooling itself is not keeping pace with the size of agent runtimes. Empirical

The paper deliberately pseudonymizes the audited products ("OpenClaw", "Nanobot", "PicoClaw", "ClawHub"); this review preserves those names exactly and makes no claim about which real software they correspond to.

Agents reach for more privilege than the task needs

Even with a safe model, the agent's tool selection can quietly expand the blast radius. Yang+ '26 defines over-privileged tool selection — choosing or escalating to a higher-privilege tool when a sufficient lower-privilege alternative exists — and measures it with ToolPrivBench (544 scenarios, 8 domains, 5 risk types, every tool independently sufficient).

The failure is not that the agent cannot solve the task, but that it solves the task through a needlessly expansive channel, increasing the potential blast radius — ToolPrivBench Yang+ '26, §Introduction

Two findings make this a substrate problem rather than a model-quality footnote. First, it is amplified by transient failure: when a low-privilege tool fails for unrelated reasons, agents escalate — "repeated failures appear to erode agents' confidence in low-privilege tools" Yang+ '26. Second, general safety alignment that slashes harmful-request scores does not transfer to least-privilege choice — and can make it worse. The fix the paper validates is targeted: a privilege-aware SFT+GRPO recipe that drops the rate (to 18.93% for a reasoning Qwen3-4B variant) while retaining 96–100% of MMLU/GSM8K/MetaTool capability.

⟡ My inference — not from the papers

Put ToolPriv next to ClawAudit and a shared shape appears: the agent's danger lives at the boundaries between components — tool→privilege, parser→executor, skill-loader→permission-gate — not inside the model's weights. "Is the model safe?" is becoming the wrong unit of analysis; "is the tool-selection policy and runtime safe?" is the question both papers are really asking. That is an inference about framing across two papers, not a claim either makes about the other.

Designing robustness in, not bolting it on

If patches fail, the constructive move is to make robustness an objective of the design itself. Yue+ '26 (AutoRAS) automates multi-agent-system design by representing a system as a sequence of symbolic primitives — jointly encoding topology and behaviour, including explicit safety primitives — compiled deterministically and searched with a GFlowNet / trajectory-balance optimizer driven by execution-derived safety signals.

▦ Smallest drop under attack, by a wide margin Empirical

Across MMLU / MS-MARCO / MATH / ProgramDev, AutoRAS posts the highest average clean accuracy (74.27%) and the smallest average performance drop under a four-vector attack suite (brain/memory/tool/agent-to-agent): −2.13%, versus −14.26% (AFlow), −18.43% (MaAS), −17.23% (G-Safeguard) Yue+ '26. Ablation: removing the textual safety signal causes the largest robustness loss (MMLU-under-attack 82.35 → 71.90). The learned policy is just 4.29M parameters over frozen API LLMs. Empirical

once security incidents occur, agentic workflows cannot be reliably protected by patch-style defenses alone, but require redesign with safety as a first-class principle — AutoRAS Yue+ '26, §Ablation

◇ Read the robustness claim at the size it was made

AutoRAS scopes its own result carefully: "our robustness claim refers to structural robustness within the evaluated agentic threat model, not immunity to all possible attacks" Yue+ '26. It evaluates four attack vectors with partial-compromise injection, and over-weighting robustness degrades clean accuracy — an explicit robustness–utility tradeoff. Like the other two safety papers here, it is a strong result inside a defined threat model, not a general guarantee; the honest reading is "this is the right place to defend," with the size of the win still benchmark-bound.

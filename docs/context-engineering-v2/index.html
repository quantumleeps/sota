State of the Art · Context Engineering for LLM Coding Agents

Steering a coding agent becomes an infrastructure problem, not a prompt

Across these 17 papers — all posted between January and June 2026 — the thing that decides whether a coding agent succeeds is less the model and more the context system built around it: what repository knowledge is codified for it, how the right code is found and fed in, how the working context is kept inside its budget, what persists across sessions, and what governs the actions it emits. The recurring move is to treat context as engineered, version-controlled, machine-targeted infrastructure — and several papers warn that doing it carelessly makes the agent worse.

This is a review of 17 sources, gathered in June 2026 by an automated arXiv-relevance search for "context engineering for LLM coding agents" (repository instruction files, harness scaffolding, context provisioning, long-horizon context, project memory, governance) and read as a sealed corpus. What recurs across these 17 papers

Six observations have enough independent support within this corpus to count as consensus here. Each links to the page that develops it.

1 · Reading the repository is the dominant cost — and the main lever Empirical

Multiple papers independently measure that gathering code context dwarfs editing and execution: read operations consume 76.1% of agent tokens SWE-Pruner '26; reading and searching are 46.5% of tokens and 56.2% of tool-use turns FastContext '26; and how well an agent selects context predicts whether it resolves the task at Pearson r=0.950 SWE-Explore '26. → Provisioning · Context Limits

2 · Codified context is moving from one hand-written file to a tiered, machine-maintained substrate Proposal

The single CLAUDE.md gives way to structured, layered, often self-maintaining context: a three-tier constitution + specialist agents + knowledge base Vasilopoulos '26, a budgeted prompt-resident "context map" discovered from execution Peek '26, a harness of instructions/skills/tools the agent rewrites from its own runs Pan+ '26, versioned skill assets with a provenance graph SkillWiki '26, and an append-only project event log PROJECTMEM '26. → Instruction Files · Memory

3 · Auto-generated instruction files are not a free win — hygiene beats presence Empirical

The corpus is openly skeptical of "just run /init." Auto-generated context files slightly reduce success and add >20% cost Gloaguen+ '26; 91 of 100 popular repos' files carry quality smells like bloat and leakage dos Santos+ '26; and stale specs that silently contradict the code are named the primary failure mode of codified context Vasilopoulos '26. → Instruction Files

4 · Decoupling exploration from solving — a clean context interface — recurs as a winning pattern Empirical

A separate component finds the context and hands the solver a compact, structured artifact, keeping search noise out of the solver's window: a trained explorer subagent returning file-line citations FastContext '26, a structure-first phase that writes an INSTRUCTION.md before editing RepoMirage '26, exploration scored as its own line-level task SWE-Explore '26, and summary-only subagent isolation in a production harness Liu+ '26. → Provisioning · The Harness

5 · Lossy compression that works for QA fails for multi-step coding; missing context hurts more than redundant context Empirical

Coding agents are unusually sensitive to fidelity loss. Implicit (embedding) compression that helps single-shot QA solves fewer SWE-bench issues than no compression at all Gelvan+ '26; missing core context lowers resolve rate more than redundant context does SWE-Explore '26; line-level pruning preserves syntax where token-level shreds it SWE-Pruner '26; and naively persisting full history hurts Peek '26. → Context Limits

6 · Governance and provenance move to the execution and contribution boundaries Proposal

Several papers shift control from inside the model to the boundary where the agent acts or contributes: a model-agnostic layer that approves/revises/blocks/escalates emitted actions Shi+ '26, a deterministic pre-action gate keyed to logged failures PROJECTMEM '26, open-source policies and provenance trailers for machine contributors Manita & Amari '26, and "governed" named as a target property of the harness itself Ning+ '26. → Governance

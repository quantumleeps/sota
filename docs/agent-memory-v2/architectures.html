Architectures of memory

What does an agent's memory actually look like in this corpus? The answers span graphs, layered stores, patch histories, latent vectors, and a fast working-memory tier — but they share a direction: memory is given a structure that is meant to change. This page maps the structural families and the working-memory layer that decides what stays in the context window.

⟡ My framing — not a paper's claim

Grouping these systems into "families" (graph / layered / patch / latent / working-memory) is my organisation of the corpus. The papers do not agree on a taxonomy; I use these buckets to make the design space legible. Chips mark the systems I am reasoning from.

Graph & connectivity memory

Two systems represent memory as a graph and treat reasoning over it as traversal rather than flat retrieval.

FluxMem models memory as a heterogeneous graph with three layers — semantic knowledge, episodic experiences, and procedural skills — and argues the agent's working context is just an activated subgraph, so optimizing context becomes topological editing: expanding under-connected links, pruning interfering ones, reshaping a unit's granularity, and distilling recurrent successful trajectories into procedural "circuits." FluxMem '26 Proposal Its ablations attribute most of the LoCoMo gain to online feedback-driven refinement and most of the web-navigation gain to offline consolidation FluxMem '26 Empirical — i.e. the two evolution stages do different jobs.

optimizing the working context is equivalent to performing targeted topological edits on the graph — FluxMem FluxMem '26, §Context as Dynamically Induced Connectivity

MemDreamer applies the same intuition to long video: a perception model streams video into a persistent, purely-textual three-tier graph (video root → super-events → macro-events, over an entity/micro-event subgraph), and a separate reasoning model answers by an agentic Observation–Reason–Action loop with navigation, search, and graph-traversal tools. MemDreamer '26 Proposal Crucially, hierarchy and the graph give complementary gains, and adding graph-traversal tools is the single largest jump in its retrieval ablation. MemDreamer '26 Empirical

Layered & hierarchical stores

Where graph systems emphasise links, two others emphasise tiers with different retention rules.

GenericAgent keeps a four-layer store — an always-on L1 index, an L2 fact layer, an L3 layer of consolidated SOPs, and an L4 raw session archive — plus a global meta-memory, injecting only meta-memory + the L1 index by default and routing to deeper layers on demand. GenericAgent '26 Proposal Verified trajectories are consolidated into SOPs and then executable code under an explicit "No Execution, No Memory" rule, so guesses and failed branches never enter long-term memory. GenericAgent '26 Proposal A memory-ablation finds a condensed memory matches a redundant one at a fraction of the tokens (165 vs 575 tokens at equal task success), so density, not volume, is what pays. GenericAgent '26 Empirical

MUSE-Autoskill splits memory by scope: a compressed short-term task context, an uncompressed long-term cross-session note store, and — its distinctive move — a per-skill .memory.md file holding lessons and usage caveats for each skill. MUSE '26 Proposal It also keeps a conversation as a DAG of nodes with a mutable active chain and immutable replayable history, so the working context can be compressed without losing recoverable state. MUSE '26 Definition

skills should not be one-off generation outputs but long-lived, evolving assets of an agent system — MUSE-Autoskill MUSE '26, §Introduction

Version-aware & patch memory

EvoArena names a failure mode the layered and graph stores mostly share: state collapse, where memory is kept as "a single latest state" and breaks when different environment versions need different behaviour. Its remedy, EvoMem, is a "git-like" append-only patch history that records only non-additive updates — each patch storing the pre-update and post-update memory, the rationale, and the triggering evidence — so memory becomes a traceable history rather than an overwrite. EvoArena '26 Proposal It helps most at the multi-step "chain" level (+3.7% chain accuracy on its benchmark) and reduces code-regression rates, but the authors are explicit that it is "not a replay buffer." EvoArena '26 Empirical

reliable agents in changing environments should treat memory as an evolving history of grounded updates, keeping the latest state connected to the prior states, rationales, and evidence that produced it — EvoArena / EvoMem EvoArena '26, §Introduction

Self-evolving retrieval infrastructure

Most systems evolve stored content but freeze the machinery that queries it. EvolveMem makes the opposite move: it exposes the entire retrieval configuration — semantic / keyword / structured top-k, fusion weights, per-category overrides, answer policy — as an action space, and lets an LLM-driven diagnosis module read per-question failure logs and rewrite that configuration in a closed "AutoResearch" loop, reverting changes that regress. EvolveMem '26 Proposal The loop even proposes parameters absent from the original space (adversarial entity-swap, query decomposition, answer verification), and replacing the LLM diagnosis with random search over the same space costs roughly a tenth of the F1 — evidence the failure logs carry real signal. EvolveMem '26 Empirical

truly adaptive memory requires co-evolution at two levels: the stored knowledge and the retrieval mechanism that queries it — EvolveMem EvolveMem '26, §Abstract

Latent & compressed memory

Not all memory in the corpus is human-readable text. JAMEL compresses each interaction step — an (observation, action) pair — into a single latent memory token (a frozen vision-language model's hidden state), aligns it into the policy's embedding space, and keeps the whole session as a soft prefix. JAMEL '26 Proposal This avoids context truncation entirely and matches cloud-model exploration depth at 2.8×–22× lower token cost JAMEL '26 Empirical — but the paper is candid about the catch.

Latent memory is usually more efficient but much harder to supervise, because its compressed vectors lack human-readable semantics. — JAMEL JAMEL '26, §Introduction

The fast tier: working memory & context management

Underneath every long-term store sits the context window — the fast, scarce tier of an agent's memory hierarchy — and two systems treat managing it as a memory problem in its own right.

TokenPilot argues that the usual context-reduction tricks (pruning text, evicting old memory) save tokens but mutate the prompt layout, which shatters KV-cache prefix continuity and triggers cache-miss penalties that "ultimately override any financial savings from text reduction." TokenPilot '26 Theory Its answer is a working-memory lifecycle: segments move active → completed → evictable, and a completed segment keeps its physical cache slots as long as its residual utility is non-zero, so cross-task document knowledge is preserved rather than re-read. TokenPilot '26 Proposal Prefix stabilization alone raises the cache hit rate from 38.7% to 79.2% on one benchmark. TokenPilot '26 Empirical

GenericAgent attacks the same tier from the density side, targeting a budget under ~30k tokens through head-tail truncation, periodic tag-level compression, FIFO eviction, and a working-memory anchor of recent turn-summaries — and reports that lower token consumption tracks better task performance, not worse. GenericAgent '26 Empirical

token consumption reflects the symptoms of an agent's context management quality, rather than the thoroughness of reasoning — GenericAgent GenericAgent '26, §Discussion

⟡ My reading of the architecture space

Two axes organise this corpus for me. The first is representation: human-readable (graphs, layered text stores, patch logs) versus latent (JAMEL's vectors). Readable memory is auditable and editable — which is exactly what the consistency work needs — while latent memory is cheaper but, by its own authors' admission, hard to supervise. The second is what evolves: most systems evolve stored content; EvolveMem is the outlier that evolves the retrieval machinery, and EvoMem evolves the edit history rather than the state. If I had to name the through-line, it is that everyone is adding a layer whose job is to decide what the memory should become — and that layer, not the store, is where these systems differ.

◇ What this page can't tell you

Every system here is evaluated by its own authors on its own chosen benchmarks; there is no shared architecture comparison in the corpus, so "which structure is best" is not answerable from these papers. WorldMemArena (on Consistency) is the closest thing to a neutral cross-design comparison, and it is multimodal-specific.

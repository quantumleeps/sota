04 · Memory Hierarchy

The context window as a cache

The boldest move in the 2026 corpus is to stop treating the context window as "the memory" and start treating it as the fast, scarce top of a hierarchy — borrowing demand paging, virtual memory, and cache management directly from operating systems. Three papers build this at three different levels.

The thesis: context is L1 cache, not memory

The framing paper states it flatly: the context window "is not memory. It is L1 cache — a small, fast, expensive resource that the field treats as the entire memory system" Mason '26 Theory. The contribution is a precise mapping of OS abstractions onto agent context:

OS conceptAgent-context equivalent
Physical memorythe context window (~200K tokens)
Virtual memorypersistent state (disk, databases)
Page faultmodel re-requests evicted content
MMUa proxy layer between client and API
Working setthe currently relevant context subset
Thrashingevict/fault cycle exceeding useful work

Source: the VM-analogy table in Mason '26 Definition. The paper proposes a four-level hierarchy (L1 generation window → L2 working set → L3 session history → L4 cross-session memory), of which only the upper levels are built and measured.

The empirical backbone is rare production data: across 857 sessions and 4.45 billion input tokens, 21.8% of tokens are structural waste, and stale tool results are reprocessed at a median 84.4× amplification Mason '26 Empirical. A deployed proxy ("Pichay") recovers context — one steady-state session went from 7% to 43% free.

Why LLM memory management is not classical VM

The sharpest theoretical insight is the inverted cost model. In classical virtual memory, keeping a page is free and faulting is expensive, so you minimize faults. With LLMs, every retained token costs money every turn, while re-fetching is cheap — so the economics flip Mason '26 Theory:

when keeping is expensive and faulting is cheap, aggressive eviction is correct by default. — The Missing Memory Hierarchy Mason '26, Discussion

With one twist: because attention cost is roughly quadratic, the paper argues eviction should become more conservative as the window fills — "the opposite of the naive instinct."

Level: the conversation log — CMV's "Git for context"

Contextual Memory Virtualisation works one level down, on the raw JSONL log. It models session history as a DAG of immutable snapshots and branches — "effectively a Git-like workflow for conversational memory" — and adds a three-pass structurally lossless trimmer that strips mechanical bloat (raw tool dumps, base64 images, metadata) while keeping every user/assistant message verbatim Santoni '26 (CMV) Proposal. Its motivating observation is the destructiveness of native compaction:

[native autocompaction] reduced 132k tokens of accumulated message state to 2.3k — a 98% reduction. — Contextual Memory Virtualisation Santoni '26 (CMV), Introduction

Trimming cut tokens by a mean 20% (max 86%) across 76 sessions Empirical, but the author is candid that "the dominant cost avoided … is not trimming but context rebuilding" — reloading an 84K-token mental model takes 10–20 turns, where a branch loads it in one prompt.

Level: the KV cache — SideQuest's self-GC

SideQuest operates at the lowest level — the GPU tensor state. Static KV-cache heuristics (H2O, SnapKV) assume past attention predicts future relevance, which breaks in agentic reasoning where "a low-importance token … may suddenly become critical … ten turns later" Kariyappa & Suh '26 Theory. Instead, the model manages its own cache: an auxiliary thread on the shared context reasons about which tool-output tokens are stale and evicts them, with no permanent token overhead to the main window Proposal.

SideQuest equips the model with a mechanism to selectively clear segments of its own KV cache, effectively allowing the model to perform its own memory garbage collection. — SideQuest Kariyappa & Suh '26, Introduction

It cuts peak token utilization 56–65% and lifts serving throughput 83.9% on an H100, for ≤2% (in-distribution) to ≤5% (out-of-distribution) accuracy loss Empirical (see Evidence).

◇ The precursor everyone cites: MemGPT

All three surveys frame this thread as descending from MemGPT's OS-style virtual memory (main context / recall / archival storage). The memory survey's caution is worth carrying forward: the Achilles' heel of hierarchical memory is orchestration, whose failures are silent — a wrong eviction yields a slightly worse answer, not an error Du '26.

⟡ My inference — not from the papers

These three aren't competitors — they tile one hierarchy at different granularities: SideQuest manages the KV tensors, CMV versions the conversation log, Demand Paging governs eviction across the whole stack. Read together they sketch a layered context OS, and each explicitly nods to the others as complementary. The open question I'd flag: every layer here is evaluated on a single-user or single-model setup, so the hierarchy is a compelling design pattern with thin external validity — promising architecture, early evidence.

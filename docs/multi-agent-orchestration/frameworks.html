01 · Orchestration Frameworks

From fixed pipelines to learned coordination

Six papers in this corpus ship an orchestration system: a main agent or controller that decomposes a task and coordinates specialized workers, tools, or models. Read together, they share one direction of travel — the coordination logic that used to be hand-wired (a chosen topology, a fixed routing rule, a static pipeline) is becoming a learned, inspectable policy conditioned on the task and, in one case, on the live serving infrastructure.

What an "orchestrator" coordinates

The frameworks differ in what they coordinate, which is worth separating before comparing them.

  • Modality-specialized sub-agents. Orchestra-o1's main agent delegates perception/action to text/image/audio/video sub-agents over a dependency graph with parallel execution F. Zhang+ '26.
  • Frozen expert models + skills. Maestro routes over a pool of frozen experts and a two-tier skill library, picking a Model@@Skill pair per step Wu+ '26.
  • Roles mapped to a human workflow. MAAD chains Analyst → Modeler → Designer → Evaluator to turn a requirements spec into architecture R. Li+ '26.
  • Reasoning modules behind a blueprint. MOSAIC composes retrieved code modules under an explicit blueprint, using the LLM "as an executor of this structured context rather than as an unconstrained code generator" Bao+ '26.
  • A whole serving stack. InfraMind coordinates planner, executor, and a deadline-first scheduler against live GPU-cluster state Kabir+ '26.
  • The coordination policy itself. AgensFlow exposes skill, role/model, and topology choices as a learnable action space over repeated runs Koenigstein '26.

The shared move: learn the policy, and make it inspectable

▦ Coordination as an online policy-learning problem Proposal

AgensFlow reframes the design question away from model choice: not "which model is best?" but "which coordination policy works under this observable task regime and operating constraint?" — and makes omission a first-class topology decision via a skip:X action, with the reward signal itself "exposed to cross-judge audit rather than treated as a fixed evaluator" Koenigstein '26. The learned routing moved off the default solver on 7 of 8 task classes, keeping the default only on the highest-cost-of-error class — risk-adaptive, not uniform cheapening.

the relevant question is therefore not "which model is best?" but "which coordination policy works under this observable task regime and operating constraint?" — AgensFlow Koenigstein '26, §Discussion

▦ Orchestration can substitute for model scale Empirical

Maestro's 4B learned orchestrator, composing frozen experts, reaches 70.1% average across ten multimodal benchmarks — above GPT-5 (69.3%) and Gemini-2.5-Pro (68.7%) — and removing the model pool costs −12.1% while removing skills costs only −2.7%: "specialized models supply the domain-specific 'brain'… skills serve as the 'eyes' and 'hands'." RL is the driver: the same workflow without it scores 52.7% Wu+ '26. Orchestra-o1 echoes this with a harness ablation — under identical tools, its orchestration lifts a GPT-5 ReAct agent from 53.9% to 72.8% on OmniGAIA, so the gain "comes from orchestration design, not just the backend" F. Zhang+ '26.

▦ Structured intermediate artifacts beat free-form generation Empirical Proposal

Two systems insert an explicit artifact between intent and execution. MOSAIC's blueprint grounds generation in retrieved modules; removing the blueprint drops its win rate to 0% on two datasets, and it holds a 100% run-success rate where baselines range 20–100% Bao+ '26. MAAD makes per-stage evaluation "an integral part of architecture generation… rather than a one-shot generation task," and its Evaluator-driven loop beats MetaGPT on architecture-quality metrics R. Li+ '26.

Infrastructure-awareness: the newest axis

InfraMind is the corpus's clearest argument that orchestration research has been ignoring the machine the agents run on.

▦ "Infrastructure blindness" Definition Empirical

Routing on static task/model features alone ignores runtime serving state. Profiling a prior router under load, preferred small models "accumulate queues exceeding 130 requests while equally capable large models sit nearly idle," producing >30 s delays on queries an idle alternative would answer in under 10 s — yet at low load 67% of large-model GPU capacity goes unused Kabir+ '26. InfraMind conditions topology, model, and reasoning depth on live queue/KV/latency telemetry, reaching up to 99.9% SLO compliance under high load where every baseline falls below 50%.

⟡ My inference — two different things are both called "orchestration" here

It is worth not over-reading the convergence. Three of these systems (Orchestra-o1, Maestro, InfraMind) orchestrate a population of distinct agents/models; two (MOSAIC, and arguably MAAD's tightly-scripted chain) are closer to a single staged pipeline with role-labelled stages and a planner/executor split. They share vocabulary — "blueprint," "main agent," "delegation" — but the coordination problem differs: routing among autonomous peers versus sequencing one workflow. I read the corpus's "learn the policy" throughline as genuinely shared, but the hardness of what is being learned is not comparable across these two groups. Chips here mark the systems I reason from, not papers that draw this distinction themselves.

◇ The shared caveat: the base model is the ceiling

MAAD states the bound directly — orchestration is a "capability amplifier" that "elevates the generation precision… of stronger LLMs, while exposing the deficiencies of weaker LLMs," and the base model's competence "establish[es] a performance ceiling for the entire pipeline, rather than a mere bottleneck that coordination strategies can easily bypass" R. Li+ '26. Every framework here is reported against strong backbones; none demonstrates that orchestration rescues a weak one.

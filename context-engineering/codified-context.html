02 · Codified Context

The artifacts & their anatomy

"Codified context" is documentation written for a machine reader. The anchor defines it precisely as "structured artifacts written explicitly for machine consumption — documents whose primary audience is an AI agent, not a developer," treated as "load-bearing artifacts that AI agents depend on to produce correct output" Vasilopoulos '26 Definition. This page covers the artifact family, the empirical anatomy of the files developers write, and the move from one flat file to tiered infrastructure.

The artifact family

The reverse-engineering of Claude Code gives the most complete map of the mechanisms, ordered (its framing) by how much context budget each costs Liu+ '26:

Instruction files

CLAUDE.md / AGENTS.md

A four-level hierarchy — managed → user → project (incl. .claude/rules/*.md) → local — loaded in reverse priority so closer-to-CWD files get more attention. Delivered as a user message, so compliance is "probabilistic rather than guaranteed"; permission rules are the deterministic enforcement layer. Liu+ '26

Skills

SKILL.md + frontmatter

A meta-tool injects skill instructions into the current context (low cost — only descriptions stay resident). Frontmatter carries allowed-tools, model override, execution context. Liu+ '26

Sub-agents

.claude/agents/*.md

Spawn a new isolated context; the markdown body is the system prompt, frontmatter sets tools/model/permissions. They return summary-only text to the parent and write separate sidechain transcripts — agent teams cost ≈7× a standard session. Liu+ '26

Tool context

MCP servers

The primary external integration and the highest-context-cost mechanism; deferred tool schemas load on demand to conserve the window. A "single unified API cannot span the full range," which is why four mechanisms coexist. Liu+ '26

Beyond Claude Code, the same artifact idea appears as Cursor's .cursor/*.mdc rule files (93% of rule-using repos place them in .cursor/) Jiang & Nam '26, GitHub Copilot's copilot-instructions.md Chatlatanagulchai+ '25b, and AGENTS.md as the cross-tool open format (60,000+ repos by the time of the controlled study) Lulla+ '26.

What they actually look like — a convergent anatomy

Four independent empirical studies, on different tools and disjoint repo samples, describe the same artifact. That convergence is itself the strongest signal in this part of the corpus.

Structure: a shallow, predictable hierarchy

CLAUDE.md files use "shallow hierarchies with one main heading and several subsections" — median 1 H1, 5 H2, 9 H3, with H4+ rare Chatlatanagulchai+ '25a Empirical. The cross-tool follow-up confirms it across Claude/Codex/Copilot (median 1 H1; H2 6–7; H3 9–12) Chatlatanagulchai+ '25b, and the independent UFMG study finds a median of 7 level-2 sections, with 36 of 328 files using no ## sections at all Santos+ '25.

Content: heavy on "how," light on "how well"

The content profile is functional and action-oriented — commands, implementation detail, architecture, tests — with a consistent neglect of non-functional requirements. The studies use different category schemes (so the numbers below are not strictly comparable cell-to-cell), but the pattern is robust across all of them.

The OSS cross-tool study reports the same hierarchy of concerns for AGENTS.md by raw counts (Conventions 50, Contribution guidelines 48, Architecture 47, Build commands 40… Security only 6) and notes "there is no established content structure yet" Mohsenimofidi+ '26. It adds a second axis the others miss — a five-way style taxonomy orthogonal to topic: descriptive, prescriptive, prohibitive ("Never commit directly to main"), explanatory, conditional Definition.

Cursor rules: traditional docs plus a new directive class

The Cursor-rules study finds developers largely "use cursor rules as an extension of traditional documentation," organised into five themes — Project, Convention, Guideline, LLM Directive, Example Jiang & Nam '26 Definition. The genuinely new class — LLM Directives (behaviour, workflow, persona, output formatting) — is the least adopted (≈50% of repos vs 84–89% for the doc-like themes), "suggesting many developers may not yet be familiar with the practice of authoring these prompt-engineering strategies." And 28.7% of all rule lines are exact duplicates, much of it reused from templates like awesome-cursorrules Empirical — motivating their call for "context transparency": don't spend the budget on what the agent already knows.

They behave like configuration code, not documentation

The largest study reframes the whole artifact class: these files "are not static documentation but complex, difficult-to-read artifacts that evolve like configuration code, maintained through frequent, small additions" Chatlatanagulchai+ '25b Empirical. Concretely: Claude files are long (median 485 words) and very hard to read — a Flesch Reading Ease of 16.6, "comparable to dense legal contracts"; 67.4% are touched in multiple commits; updates come as bursts of additions (median +57 words/commit) with negligible deletions. The OSS study corroborates the evolution shape — of 155 AGENTS.md files, half never changed and most changes "add" or "modify" instructions rather than restructure Mohsenimofidi+ '26.

▦ The recommended mental model — and a new debt

"Developers should… treat files like CLAUDE.md or AGENTS.md with the same rigor applied to a Dockerfile or CI/CD workflow." The same paper names the failure mode of not doing so — context debt: low readability plus accreting length make files "unmaintainable and opaque to human collaborators." It proposes CODEOWNERS-style governance, CI linters that check documented build commands against package.json/Makefile, NFR scaffolds, and semantic-category-aware retrieval. Chatlatanagulchai+ '25b

The single-file ceiling

Every study above describes a single (or near-single) flat file, and the adoption data confirms that is the norm — most adopters keep one modest guidance file Robbes+ '26. The anchor's premise is that this does not scale:

a 1,000-line prototype can be fully described in a single prompt, but a 100,000-line system cannot.— Codified Context Vasilopoulos '26, Introduction

From file to infrastructure — the anchor's three tiers

The anchor generalises the manifest into a tiered architecture, built solo over 283 Claude Code sessions on a 108,256-line C# multiplayer simulation. The whole thing reached ≈26,200 lines across 54 files — 24.2% of the codebase, which the paper is careful to call "project-specific, not a target or finding" Vasilopoulos '26 Proposal Empirical.

Tier 1 · hot memory

The Constitution

≈660 lines, always loaded. Code-quality standards, naming, build commands, architecture summaries, known failure modes, and orchestration / trigger-table protocols. A scaled CLAUDE.md whose governing constraint is conciseness.

Tier 2 · per-task

Domain specialists

19 sub-agent specs (115–1,233 lines each, ≈9,300 total). Over half of each is embedded domain knowledge — they are "domain-priming mechanisms," not just routers. Emerged from observed failure patterns, not upfront design.

Tier 3 · cold memory

Knowledge base

34 markdown files (≈16,250 lines), one subsystem each, written for AI consumption (explicit patterns, file paths, parameter names, failure-mode tables). Retrieved on demand.

Two pieces of connective tissue make it a system rather than a folder of docs:

  • An MCP retrieval service (≈1,600-line Python server) exposes five tools — list_subsystems(), get_files_for_subsystem(), find_relevant_context(), search_context_documents(), suggest_agent() — though retrieval is still keyword substring matching, with semantic retrieval flagged as future work Vasilopoulos '26.
  • Trigger tables + a context-drift detector. Tasks route to specialists on observable signals (the files being modified); a session-start hook diffs recent Git commits against the subsystem→file map and warns when source changed but specs did not — a direct mechanical answer to staleness.

The reported behaviour: of 757 classifiable agent invocations, 57% went to project-specific specialists (most-used: code-reviewer 154×, network-protocol-designer 85×); over 80% of human prompts were ≤100 words (pre-loaded context replacing in-prompt explanation); and the most-referenced spec (the save system, 74 sessions) saw "no save-related bugs" across those sessions. The paper frames its own contribution against codebase indexers precisely:

These systems index code; the codified context infrastructure indexes knowledge about code — design intent, constraints, and failure modes.— Codified Context Vasilopoulos '26, Related Work

◇ How much to trust this

This is a single-developer, single-project, observational experience report — by the author's own statement "a systems paper and experience report," with no controlled comparison and confounds (e.g. the developer's own growing experience) unisolated. Treat the architecture as a well-documented existence proof and the metrics as descriptive, not causal. Vasilopoulos '26

The "living context" forms

Two systems push codified context from written-once toward self-maintaining:

ACE argues contexts "should function not as concise summaries, but as comprehensive, structured playbooks," represented as itemised bullets each carrying a unique ID and helpful/harmful counters, updated by incremental deltas rather than monolithic rewrites, with a Generator–Reflector–Curator loop and periodic de-duplication Zhang+ '26 Proposal. This is the operational counter-argument to "summarise/compress" and the clearest blueprint for a context store that grows safely (see Evidence for its gains and Failure Modes for the collapse it prevents).

Confucius backs working memory with a file-system tree of markdown leaf nodes plus a dedicated note-taking agent that distils each session into persistent notes — including "hindsight notes that capture failure modes," indexed by error message and stack trace for retrieval in future sessions Wong+ '25 Empirical. Notably it separates the channel the agent sees (AX) from the human-facing trace (UX), warning that conflating them "[causes] context bloat, spurious anchors, and brittle debugging."

⟡ My read: three datable layers of the same idea

Reading these together, codified context has moved through three forms in roughly two years: (1) the flat manifest (what the file studies measure), (2) tiered, retrievable infrastructure (the anchor), and (3) self-evolving stores (ACE, Confucius notes). They are not competitors so much as increasing degrees of automation over the same substrate — and most teams in the wild are still firmly at layer 1. The Synthesis page argues these should be unified rather than chosen between.

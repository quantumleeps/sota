06 · Limitations

Open problems the papers name themselves

These are the failure modes and caveats the corpus raises about its own program — grouped into families. They are notable because they come from inside the work: the same papers proposing spec-driven workflows are candid that the evidence is early, the artifacts decay, the gates can be gamed, and the hardest step — getting the spec right — is itself unsolved.

1 · The evidence is early and structurally thin

◇ Proposals outnumber measurements

Nine of the 14 papers are surveys, taxonomies, or position/vision pieces; even the empirical ones lean single-model and single-domain. Hasanli's TDD-governance evaluation is explicitly "preliminary" with no controlled metrics yet Hasanli+ '26; de Macedo's framework scores come from a single rater with no inter-rater reliability de Macedo '26; Mandl rests on one exploratory case "not intended to validate the model empirically" Mandl+ '26; Grynets used 3 replications on one domain/model Grynets+ '26; Monperrus and Koch are explicitly syntheses, not new studies Monperrus '26 Koch '26.

◇ No benchmark for the whole process

Multiple papers note there is no benchmark for spec-driven development as a process — SWE-bench-style suites assume the spec is correct and test only implementation. de Macedo lists "lack of benchmarks for the complete process" as a headline risk de Macedo '26; Hamblin built SpecBench precisely because existing benchmarks "assume specifications are correct and complete" Hamblin+ '26; Bhati calls for evaluation "beyond SWE-bench" Bhati '26.

2 · Specs and context artifacts drift, decay, and can backfire

⚠ Drift between spec and code; "living artifacts" rot

de Macedo names "drift between spec and code" and the maintenance of "living artifacts" as recurring risks, and warns context absence "produces functional hallucinations: code that compiles, but violates implicit contracts" de Macedo '26. Taghavi's whole premise — "context blindness" — is that artifacts go stale against the repository they target Taghavi+ '26.

⚠ More context is not always better

Koch's "minimal-context principle" exists because repository instruction files can reduce task success and raise cost "when imposing unnecessary or mismatched requirements" — context must be short, current, non-contradictory, and tied to executable feedback, not maximal Koch '26.

3 · Correlated reviewers and a new attack surface

⚠ When the same model writes and reviews, blind spots correlate

Monperrus flags that having one model family generate and review code creates correlated blind spots and silent false-negative approvals, and that prompt injection is "a qualitatively new attack surface that does not exist for human reviewers" with "no fully solved defences" Monperrus '26. Alenezi independently observes emerging "closed loops" where an agent's output is reviewed mainly by a same-provider bot, with trusted security review still substantially rule-based Alenezi '26. The risk: automated acceptance that looks like assurance but shares the generator's failure modes.

4 · Security and permissions are mitigated, not solved

⚠ An explicit security spec still left 36/221 tests failing

Grynets' best condition — an application-specific security model — still failed 36 of 221 hidden tests, and the authors are explicit it is "a security-requirement operationalization layer, not… a guarantee of secure code"; hidden black-box testing remains necessary Grynets+ '26. Koch cites a stress-test where permission-gate assumptions failed under ambiguous state-changing scenarios — especially when equivalent effects were reachable via file edits instead of shell commands Koch '26. De La Cruz names the structural gap: "governance and provenance debt" when agents get tool/repo access without identities, logging, and rollback De La Cruz '26.

5 · Orchestration cannot exceed the base model

◇ The base LLM is a ceiling, not a bottleneck

Li found multi-agent orchestration acts as a "capability amplifier" that exposes weaker models' deficiencies — the base LLM's competence sets a "performance ceiling for the entire pipeline" Li+ '26. Consistent with this, Taghavi's context-grounding helped least where defects sat in deep application/library logic weakly exposed by local tests Taghavi+ '26, and Suresh's Spec-Agent struggled to find the strongest contract for the most complex functions, "possibly indicating a ceiling in current techniques" Suresh+ '26. Process discipline reorganises capability; it does not manufacture it.

6 · The upstream step — getting the spec right — is itself unsolved

◇ Specification-level reasoning sits at 44.4%

The paradigm assumes a good spec exists, but the corpus's one direct measurement of that capability shows agents below 45% at identifying spec deficiencies before code — and SpecBench so far covers only deficiency identification, not spec revision, with LLM-only judges and gold labels pending human validation Hamblin+ '26. Mandl's caution follows: AI-generated requirements are "drafts that require systematic review and domain-specific approval," not finished specs Mandl+ '26.

7 · Long-horizon debt and maintenance are unmeasured

◇ The technical-debt hypothesis

Bhati raises the worry that agents bias toward producing more code and local fixes (global redesign being token-expensive), potentially inflating long-term maintenance burden — and notes no longitudinal repository-health studies exist yet Bhati '26. De La Cruz and Alenezi echo the call for 6/12/24-month studies of AI-generated code's effect on debt and maintainability De La Cruz '26 Alenezi '26.

⟡ The cross-cutting tension I see

A specification in this corpus must be two contradictory things at once: machine-checkable (so gates, fuzzers, and deterministic engines can enforce it at agent throughput) and human-meaningful (so it actually captures intent and someone can be accountable for it). The verification cluster optimises the first; the intent-centric cluster insists on the second; and almost nothing in the corpus measures whether a spec that satisfies the machines still means what the humans intended. Until there are process-level benchmarks and longitudinal studies, "spec-driven development" rests on a plausible mechanism whose end-to-end payoff — and whose failure rate when the checkable spec and the intended spec diverge — is, in this corpus, asserted more than shown. This is my reading, not a claim any single paper makes.

Transcript

Steering a coding agent is becoming an infrastructure problem, not a prompt. That's the thread running through this review.

This is a review of seventeen papers, all posted between January and June of twenty twenty-six, and gathered in June by an automated search for work on context engineering for coding agents. So everything you're about to hear is a reading of this particular set of papers — a recent, search-shaped sample of mostly un-peer-reviewed preprints. It is not a complete survey of the field, and nothing here is a claim about where the field as a whole is heading.

Here's how this narration goes. I'll start with the handful of ideas that recur across the corpus. Then I'll walk six themes in order — instruction files, finding context, managing context, the harness, memory, and governance. Then the evidence, spoken honestly. And at the end, clearly marked, my own reading across the papers.

What recurs across the corpus

A few positions show up again and again, with enough independent support across these papers to count as consensus — at least here.

The first, and the firmest: reading the repository is the dominant cost, and the main lever. One system, called SWE-Pruner, measured that read operations consume about seventy-six percent of a coding agent's tokens — far more than editing or running code combined. Another, FastContext, found that reading and searching take up almost half the agent's tokens and more than half of its tool calls. And a benchmark called SWE-Explore found that how well an agent selects context predicts whether it solves the task, with a correlation of zero point nine five. Three independent measurements, the same conclusion: context is where the budget goes, and where the wins are.

Second: codified context is shifting away from a single hand-written file toward something tiered, structured, and increasingly machine-maintained.

Third — and this is the corpus arguing with itself — auto-generated instruction files are not a free win. Quality and upkeep matter more than mere presence.

Fourth: a pattern keeps recurring of decoupling the search for context from the act of using it, and passing a clean, structured artifact between the two.

Fifth: lossy compression that works for question answering fails for multi-step coding. And missing context turns out to hurt more than redundant context.

And sixth: governance and provenance are moving to the boundary — where the agent acts, and where it contributes.

Instruction files and codified context

First, the instruction files themselves — the AGENTS-dot-M-D and CLAUDE-dot-M-D files that have become the canonical artifact of context engineering.

This is where the corpus disagrees most usefully. On one side, an experience report from Aristidis Vasilopoulos describes building a hundred-thousand-line codebase with a coding agent as the only code generator, and growing a three-tier codified-context infrastructure — an always-loaded constitution, nineteen specialist agents, and an on-demand knowledge base — until machine-targeted documentation was about a quarter of the entire codebase. He names the central danger plainly: stale specifications. Agents trust documentation absolutely, and out-of-date specs cause silent failures.

On the other side is the sharpest skeptic in the corpus. Gloaguen and colleagues, at E-T-H Zurich, ran a controlled evaluation and found that auto-generated context files actually reduced task success slightly, while adding over twenty percent to inference cost. Developer-written files helped only marginally. And crucially, it was not a compliance failure — the agents did follow the instructions; the files just didn't help, mostly because they duplicated documentation the agent already had. Their recommendation was blunt: skip the auto-generated files, and keep the human ones minimal.

A third paper, from dos Santos and colleagues, explains part of why. Mining the hundred most-starred repositories that ship these files, they found at least one quality problem — they call them configuration smells — in ninety-one of a hundred files. The most common was including rules a linter already enforces, then sheer bloat, then stuffing rarely-used instructions into the always-loaded file.

Here I'm going slightly beyond what any single paper claims. My reading is that these results don't actually conflict. The harmful files are the auto-generated, always-loaded, redundant, unmaintained ones. The useful context is curated, loaded on demand, and kept in sync with the code. The corpus isn't really split on whether to codify context. It's split on presence versus curation.

Finding and provisioning context

Next, finding the context in the first place.

Before a coding agent can edit, it has to locate the code that matters — and three papers treat that search as a first-class, measurable problem. FastContext found that editing only begins around turn eight of a trajectory; the median agent runs six rounds of exploration before its first edit. A benchmark called RepoMirage showed that strong scores can be shallow: more than half of one model's solved cases touched only a single file. And when they perturbed the repositories in ways that preserved behavior but broke the shortcuts, average success fell from about sixty-seven percent down to twenty-five on their hardest tasks. The agents responded by opening more files — but understanding didn't follow. They named it exploration drift: access without comprehension.

The fix all three converge on is the same. Pull exploration out of the solving loop, and hand the solver a compact, structured artifact instead of a trail of search output. FastContext trains a separate explorer that returns only a block of file paths and line ranges — improving success by up to five and a half points while cutting the main agent's tokens by up to sixty percent. RepoMirage's version writes an intermediate instructions file before solving. Same idea, two forms.

Keeping context inside the budget

Now, keeping that context inside the budget.

A long coding task floods the window with file contents and logs, and the tempting fix is to compress. But the corpus's sharpest negative result, from Gelvan and colleagues at JetBrains, shows the danger. They compressed agent context into dense embeddings. On single-shot question answering, it worked. On multi-step coding, it solved seven SWE-bench issues — versus nineteen for no compression at all — and it had more room to work, not less. The reason is mechanical. Small reconstruction errors, like a slightly wrong file path, are harmless once, but they compound across steps, and a coding agent's exact-match edit tools then fail.

The opposite reflex — just keep everything — fares no better. A system from M-I-T called Peek showed that persisting the full history inflated tokens roughly ten times, for no gain, and sometimes a loss.

What worked instead was selective, structure-preserving reduction. SWE-Pruner trims a file read down to the relevant lines, preserving syntax where token-level compression shreds it. Peek keeps a small, curated map of orientation knowledge — and found that the presence of the map mattered more than its size.

Here's my own reading of these three together. The lesson isn't "compress the context." It's: protect the load-bearing tokens, drop the rest, and never lossily re-encode what a tool will match exactly. A coding agent is a brittle, exact-match machine downstream of its context, and this corpus keeps rediscovering that.

The harness as substrate

Next, the harness — the whole software layer around the model.

A framing survey from Ning and colleagues argues that code in agent systems is no longer just output; it is the operational substrate — executable, inspectable, and stateful — and that reliability is bottlenecked there, not only in the model's reasoning. They name a discipline beyond prompt and context engineering, and call it agentic harness engineering.

That abstraction gets concrete in a source-level reading of Claude Code, by Liu and colleagues. They estimate that only about one and a half percent of the codebase is the actual decision loop — a simple while-loop that calls the model and runs tools. The other ninety-eight percent is harness: context assembly, a five-layer compaction pipeline, a deny-first permission system, summary-only subagents. Their position is that as frontier models converge, that surrounding harness becomes the real differentiator.

And if the harness is the differentiator, it becomes something to optimize. Pan and colleagues showed an agent rewriting its own harness — its instructions, skills, and tools — from its own past trajectories, with no ground-truth grading, raising one benchmark's pass rate from fifty-nine to seventy-eight percent in a single offline pass. Striking — though it's one round, one model, judged by the agent itself.

Persistent project memory

Next, what survives across sessions.

A coding agent is stateless between sessions; it re-derives the same project context every time. A system called PROJECTMEM, from the University of Utah, estimates that costs five to twenty thousand tokens per session, and answers it with an append-only, plain-text event log — plus a deterministic gate that warns the agent before it repeats a fix that already failed. They call it memory-as-governance: memory that acts on the next action, rather than only answering.

Two other papers persist differently — one a memory-guided documentation generator, the other, SkillWiki, treating agent skills as versioned, governed assets with a provenance graph. And here's the striking thing they share. All three reject opaque vector databases in favor of legible, auditable, append-only structure.

That choice, I think, is not an accident. It's the same logic as the compression result. A coding agent needs recall it can trust exactly, not recall that is merely similar. Memory you can read, diff, and review beats memory that is fuzzy.

Governance and provenance

And finally, governance — control at the boundary where agents act and contribute.

A system called the Organizational Control Layer, from Shi and colleagues, intercepts an agent's actions before they execute, and either approves, revises, blocks, or escalates them, with an audit trail. In their setting, it lifted compliant success from twelve percent to ninety-six. But — and this matters — their setting was adversarial buyer-and-seller negotiation, not coding. So read that as a transferable mechanism, not a coding result.

On the social side, Manita and Amari studied how open-source projects govern AI contributors, and put their finger on the core asymmetry. AI lowers the cost of producing contributions, but it does not lower the cost of trustworthy human review. The dimension nobody governs, in any policy they examined, is maintainer review capacity.

Governance is the property the rest of the corpus implies but rarely measures — and I'd hold it loosely, because the strongest numbers come from outside coding entirely.

The evidence, spoken honestly

So what was actually measured, and how well?

The firmest ground is that context cost-and-quality finding. Three independent measurements agree that reading dominates the token budget, and that context quality tracks success. The skeptical instruction-file results also reinforce each other, across two separate groups.

The thinnest ground is almost everything about persistent memory, codified infrastructure, and governance. Those rest on single-developer self-studies, on demos without baselines, or on that non-coding domain. They're good demonstrations of a mechanism — they are not effect sizes. And one caution runs under several of the benchmark wins: these are twenty twenty-six frontier models, being tested on public repositories they may well have seen in training.

My own reading

Everything up to here was the papers. This last part is my own reading across them — inference, not findings, and scoped to these seventeen sources.

I'd organize the whole corpus as a stack. A durable spine — a minimal, human-curated instruction file. A provisioning layer that mints task-scoped context objects just in time. A working-context manager that reduces without destroying fidelity. A persistent memory that is legible and auditable. And a governance boundary that enforces deterministically, outside the model. All of it wrapped by a harness that, increasingly, edits itself.

If I had to name the bets this corpus supports: keep the always-loaded context minimal, and earn every token. Decouple finding context from using it. Optimize for the fidelity of the relevant subset, not the compression ratio. Prefer legible persistence over vector recall. And put enforcement at the boundary — deterministic, outside the weights.

And the bets I would not place, on this evidence: that auto-generated context files are useless — too strong; the studies are small and Python-only. That self-optimizing harnesses are production-ready — not on one self-graded round. Or that any single architecture has won. The corpus supports the principles. It does not yet crown an implementation.

Close

So that's the review. Seventeen papers, the first half of twenty twenty-six, read as one sealed corpus — not the field, just this set. The clearest signal across them is that steering a coding agent has become a question of the context system you build around it: what you codify, how you provision it, how you keep it inside the budget, and what governs what the agent does with the result.

07 · Adoption

Practice in the wild

Two large empirical studies bracket reality. Coding agents have spread to roughly a quarter of active GitHub projects in about a year — but deliberate context files remain a minority practice, and the developers who use agents well do so by controlling them, not vibing.

How fast — adoption at scale

The first large-scale adoption study estimates 22.20–28.66% of ≈128k active GitHub projects had adopted a coding agent within about a year, detected through their traces — config files, co-author trailers, branch/label conventions Robbes+ '26 Empirical.

The texture matters: adoption rises with project size/activity but falls with age (younger projects adopt ≈3× more); organisations adopt more than average (Microsoft near-double); and agent commits skew to features (≈35.7% feat vs a ≈17% human baseline) and are larger. The inflection points track product releases — late Feb 2025 (Claude Code), mid-May 2025 (Codex/Gemini/Jules) Robbes+ '26. The authors' framing is that the artifacts this whole thread studies are now "the primary observable signature of agent adoption":

coding agents leave abundant traces in software repositories, in the form of configuration files, summaries of knowledge they acquired, or even explicit metadata.— Agentic Much? Robbes+ '26, Introduction

What they write — guidance files in the wild

Agent adoption ≠ deliberate context engineering. The OSS study found only 5% of 10,000 scanned repos had adopted any context-file format Mohsenimofidi+ '26 Empirical, and the adoption study confirms the shape: most file-adopters keep a single modest guidance file (>85% have ≤5), while a sophisticated minority maintain >1,000-line, frequently-edited guidance — and about a quarter never update their file at all Robbes+ '26.

⟡ My read of the gap

There is a wide adoption gap between using an agent (≈25% of projects) and engineering its context (≈5% of repos, mostly one flat file). The tiered-infrastructure and self-evolving-context work on the Codified Context page is, in practice, describing what a small leading edge does — the median project is several rungs below it. That gap is the addressable opportunity, and also a caution against over-reading the frontier as the norm.

How they work — control, not vibe

The flagship behavioural study (13 observed + 99 surveyed professional developers) is unambiguous: "professional developers do not vibe code. Instead, they carefully control the agents through planning and supervision" Huang+ '25 Empirical. Concretely, they execute plans piecewise — on average asking the agent to work on just 2.1 steps at a time, never more than 5–6 even with 70-step plans — and they prize "clear context and explicit instructions."

I do everything with assistance but never let the agent be completely autonomous — I am always reading the output and steering.— survey respondent S83, Don't Vibe, They Control Huang+ '25

The study documents codified context in active use: user rules, plan files, and explicit CLAUDE.md / SPECIFICATION.md / TASKS.md workflows, a knowledge/ directory of structured docs, and MCP servers (e.g. Context7) to inject post-cutoff documentation. Its empirically grounded top-10 prompt context-type inventory is, in effect, a checklist of what developers feel they must supply:

UI/design term · technical term · domain object · reference to input file · specific library/API · interaction · new feature/requirement · reference to a plan step · reference to output file · purpose of feature

top-10 of 43 observed context types Huang+ '25

Sentiment is positive but supervised: agent-suitability rated 4.73/6, enjoyment 5.11/6; developers modify agent code about half the time; no respondent said agents could replace human decision-making. SWE expertise was cited as critical to using agents well.

One rung up, the framework-practices study shows the systems themselves are assembled, not adopted whole: 96% of top agent projects compose ≥2 frameworks, and MCP — while productivity-positive (one report cited ≈+20% engineer productivity) — brings real context overhead Wang+ '25 Empirical.

What developers say is hard

The suitability data is a candid map of the frontier. Agents are seen as suitable for repetitive/scaffolding/well-defined tasks and clearly unsuitable for business logic, legacy integration, and high-stakes/security-critical code. Two items bear directly on codified context Huang+ '25:

  • Planning / architecture is "controversial" (13 say suitable : 23 say not) — the layer most people still want to own.
  • Long-context sessions / between-task consistency is a weak spot (4 suitable : 8 not) — i.e. the agent forgetting across tasks is a felt pain, which is exactly the problem persistent codified context is meant to solve.

▦ The demand signal

"Between-task consistency" being a named weakness, while developers simultaneously maintain CLAUDE.md/SPECIFICATION.md/knowledge-dir workflows by hand, is the clearest user-pull evidence in the corpus for durable, persistent codified context. The future-work section explicitly calls for "better interfaces for the planning step." Huang+ '25

Adjacent: authored context beyond coding

For breadth, the education paper shows the same authoring pattern in a non-coding domain: each agent in a teaching simulation is given an explicit goal/role/system prompt, structured tagged output is parsed by "Concept" fields, and limited windows are handled by a hierarchical "summary of summaries" Mollick+ '24 Proposal. The connection to coding agents is conceptual, not direct — but it reinforces that authored, role-scoped context plus structured I/O and compaction is a general pattern, not a coding-only one.

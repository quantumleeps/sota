04 · The Harness as Substrate

Context engineering grows into harness engineering

Three papers zoom out from individual artifacts to the whole software layer around the model — the harness. One frames it as a discipline, one dissects a production harness from its source code, and one shows an agent rewriting its own harness from experience. Across them, context engineering is recast as one mechanism inside a larger engineered, stateful substrate.

The thesis: reliability lives in the harness, not only the model

The framing survey defines the object precisely Definition Ning+ '26:

…the software layer that surrounds an LLM with tools, APIs, sandboxes, memory, validators, permission boundaries, execution loops, and feedback channels, thereby turning a stateless model into a functional agent. — Code as Agent Harness Ning+ '26, §2

Its central claim is that code in agent systems is not just generated output but the operational medium — executable, inspectable, stateful — and that autonomy is gated by this layer Theory: "the bottleneck of autonomy is not only the reasoning ability of the base model, but also the reliability of the system that connects model outputs to long-horizon actions and persistent states." The survey explicitly names Agentic Harness Engineering as a discipline beyond prompt and context engineering, with a target property set for future systems: executable, inspectable, stateful, governed Ning+ '26.

It also reframes the artifacts from earlier pages as harness state: AGENTS.md/CLAUDE.md are "persistent, inspectable, version-controlled artifacts the agent can consult before acting," planning is lifted from an ephemeral prompt into a filesystem-backed control object (a PLAN.md that "can be reviewed by humans, versioned with Git, consumed by subagents"), and memory is "not simply a larger context window or a vector database — it is a state-management layer" Proposal Ning+ '26.

Anatomy of a real harness: what the 98.4% does

The abstraction gets concrete in a source-grounded reading of Claude Code (TypeScript v2.1.88, ~512K lines) Liu+ '26:

The core of the system is a simple while-loop that calls the model, runs tools, and repeats. Most of the code, however, lives in the systems around this loop. — Dive into Claude Code Liu+ '26, Abstract

Estimated share of the codebase that is AI decision logic; the other 98.4% is operational harness. Liu+ '26

Sequential compaction layers run before every model call — "no single compaction strategy addresses all types of context pressure." Liu+ '26

Extension mechanisms ordered by context cost: hooks (zero) → skills (low) → plugins (medium) → MCP (high). Liu+ '26

The harness is where context engineering actually happens: a documented context-assembly order (system prompt → env/git → CLAUDE.md → path-scoped rules → memory → tool schemas → history → compaction), a four-level CLAUDE.md hierarchy with lazy path-scoped rule loading, deny-first permissions, and summary-only subagent isolation (agent teams cost ~7× the tokens of a plan-mode session) Empirical Liu+ '26. The paper's position is that as frontier models converge, this surrounding harness — not the decision scaffolding — becomes the principal differentiator.

⚠ A predicted cost of the harness's own design

The same analysis warns (as an architectural prediction, not a measurement) that bounded context plus subagent isolation should produce more pattern duplication and convention drift, because "good local decisions can produce poor global outcomes when the model lacks global context" Liu+ '26 Proposal. The very mechanisms that keep context bounded can fragment the agent's view of the codebase.

The harness as an optimization target — even self-optimizing

If the harness is the differentiator, it becomes something to engineer and improve. Pan+ '26 defines the harness as "a persistent collection of tools, prompts, and skills" materialized as a directory of Markdown instruction/skill files plus executable tool scripts — and then optimizes it from the agent's own past trajectories, with no ground-truth labels, in a single offline pass (Retrospective Harness Optimization) Proposal:

an agent's own trajectories already contain the signal needed to improve it, since re-solving past tasks and comparing the outcomes exposes where the harness fails and what would fix it. — Evolving Agents in the Dark Pan+ '26, Conclusion

Using self-preference (within-trajectory self-validation + cross-trajectory self-consistency), one round raised SWE-Bench Pro pass rate from 59% to 78% without any external grading — beating memory-only methods (Dynamic Cheatsheet, ReasoningBank, Sleep-time Compute) and matching a validation-feedback optimizer at ~3× less compute Empirical Pan+ '26. The improvements are concrete context artifacts the agent wrote for itself — e.g. a repair-verify tool that locates Go's non-standard toolchain path and strips cache dirs from the final diff. Distinguishing this from memory-only approaches is the paper's core positioning: it edits the full harness (tools and instructions), not stored memory alone.

⟡ My inference — not from the papers

Stacking these three, the corpus is quietly relabeling its own subject. The instruction files, provisioning artifacts, and context-budget machinery of the previous pages are, in this framing, all harness state — and the survey says so outright by listing context engineering as one mechanism under Agentic Harness Engineering Ning+ '26. My read is that the unit of engineering is shifting from "the prompt/context" to "the on-disk substrate the agent reads, writes, and now rewrites," and that Pan+'s self-optimizing harness Pan+ '26 is the same idea as the self-maintaining context map and skill base elsewhere in the corpus. The catch is that a substrate the agent edits itself needs exactly the governance the next pages describe — Ning+ lists "self-evolving harnesses without regression" as an open problem, and Pan+ itself warns that harness updates distilled from compromised trajectories could entrench bad behavior. Self-improvement and governance are the same frontier. This is my synthesis, not a claim any paper makes whole.

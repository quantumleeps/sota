The working-memory toolkit

How practitioners actually manage a context window mid-run. By 2026 the toolkit has consolidated into a small set of composable levers — compaction, tool-result clearing, external memory, just-in-time retrieval, sub-agent isolation — plus a newer idea: let the context itself evolve. Much of this page is industry guidance; it is labelled as such.

◇ Read these as practices, not proofs

The lab writeups in this section are informed engineering guidance, not controlled studies. Their token/latency figures come from single illustrative demo runs; they show what a primitive does, not how much it helps on average. The decision frameworks are explicitly offered as "hypotheses to test on your own agent" Anthropic '26 (Cookbook).

Three primitives that compose, not compete

The Anthropic cookbook draws the cleanest distinction between the three first-party context-management primitives, separating them on three axes — scope of operation, lossiness, and persistence Anthropic '26 (Cookbook) Definition:

PrimitiveScopeLossinessPersistenceUse when…
Compactionwhole transcript → summarylossy by designin-sessionbloat is dialogue/reasoning that can't be re-fetched
Tool-result clearingsub-transcript (drops payloads)lossless if re-fetchable; zero inferencein-sessionbloat is large, re-fetchable tool output
Memory tool (file-backed)moves info out of the windowlossless on what was savedcross-sessionknowledge must survive into a new session
compaction compresses the whole window when it grows too large, clearing drops stale re-fetchable data inside the window, and memory moves information out of the window so it survives across sessions. — Anthropic Cookbook Anthropic '26 (Cookbook), "How the Three APIs Map to the Problem"

The cookbook's reframing of the decision is the durable takeaway: not "should I use all three?" but "which of the three problems does my workload actually have?" Proposal. Its demo run is illustrative only — clearing held peak context to 173,137 tokens versus a 335,279-token baseline, and compaction probed 3/3 high-level facts preserved but 0/3 obscure appendix specifics Anthropic '26 (Cookbook) Empirical (single demonstration trajectory, not a benchmark).

Just-in-time retrieval over up-front stuffing

The retrieval stance has shifted. Rather than embedding and pre-loading everything, agents keep "lightweight identifiers (file paths, stored queries, web links)" and load data at runtime via tools Anthropic '25 Proposal. This enables progressive disclosure — discovering context incrementally through exploration — and a hybrid default (Claude Code keeps CLAUDE.md in context up front, but uses glob/grep to pull files just-in-time, bypassing stale indexes).

Sub-agent isolation

A specialized sub-agent works in its own clean context window and "returns only a condensed, distilled summary of its work (often 1,000–2,000 tokens)" while having consumed "tens of thousands of tokens or more" internally Anthropic '25 Proposal. The general principle — isolate work in a separate context, return only the distillate — recurs at the systems level too (see SideQuest's parallel auxiliary thread on the Memory Hierarchy page).

External memory as structured notes

The cross-session lever is note-taking: the agent writes notes "persisted to memory outside of the context window" and pulls them back later Anthropic '25 Definition. OpenAI's cookbook turns this into a full lifecycle for personalization — a local-first state object (structured profile + free-form notes), with four moves OpenAI '26 Proposal:

  • Distill candidate notes during a run via a save_memory_note(...) tool;
  • Consolidate asynchronously at session end — dedup, conflict-resolve, and forget;
  • Inject curated slices into the system prompt under explicit precedence rules ("the user's latest message … overrides everything");
  • Forget on purpose — promote durable notes, prune the rest.
Forgetting is not a bug—it is essential. — OpenAI Cookbook OpenAI '26, "Memory Consolidation"

The same source flags the cost: because notes are injected straight into the system prompt, "memory systems are a high-value attack surface" needing guardrails at every stage OpenAI '26 — a theme the Failure Modes page returns to.

The newer idea: let the context evolve

ACE pushes past static curation: treat the context as an evolving playbook of itemized strategies that "continuously accumulate, refine, and organize" over time, updated by a Generator/Reflector/Curator loop with small, localized delta edits rather than monolithic rewrites Zhang+ '25 (ACE) Proposal. Its central warning is against over-compression — context collapse:

at step 60 the context contained 18,282 tokens … but at the very next step it collapsed to just 122 tokens, with accuracy dropping to 57.1. — Agentic Context Engineering Zhang+ '25 (ACE), §Background

ACE argues that, unlike humans, LLMs benefit from long, detailed contexts and "can distill relevance autonomously," so domain heuristics should be preserved rather than summarized away — and reports +10.6% (agents) / +8.6% (domain) gains with large adaptation-cost reductions (see Evidence) Zhang+ '25 (ACE) Empirical.

⟡ My inference — not from the papers

Put the levers on one axis and a design rule falls out: curate by default, compress only when re-fetch is impossible. Clearing (re-fetchable) and just-in-time retrieval keep fidelity for free; ACE argues against compressing what the model could distill itself; the cookbook says reach for lossy compaction only when the bloat is reasoning you can't reconstruct. The corpus doesn't state this rule as such — it is my reconciliation of the cookbook's decision framework Anthropic '26 (Cookbook) with ACE's anti-collapse stance Zhang+ '25 (ACE).

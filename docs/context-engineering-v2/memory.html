05 · Persistent Project Memory

What survives across sessions

A coding agent is stateless between sessions: it re-derives the same project context and re-attempts fixes that already failed. Three papers build persistence — an event-sourced project log, a memory-guided documentation generator, and a versioned skill base — and they share a striking choice: legible, provenance-linked structure over opaque vector recall.

The statelessness tax

PROJECTMEM '26 estimates that re-establishing context in a stateless agent costs 5,000–20,000 tokens per session, and argues the bottleneck is "missing project memory," not model capability Empirical. Its fix is an append-only plain-text event log (five typed events: issue, attempt, fix, decision, note) deterministically projected into AI-readable summaries served over MCP — cutting per-session context to ~800–1,500 tokens, an estimated >50% reduction Empirical. The defining property is determinism:

the summary can be rebuilt from scratch at any time and can never silently diverge from history — the property that makes the plain-text substrate trustworthy as an audit trail. — PROJECTMEM PROJECTMEM '26, Implementation

Memory that acts, not just answers

PROJECTMEM's distinctive contribution is a deterministic pre-action judgment gate keyed to the project's own logged failures — precheck_file() returns "you tried this 2 days ago — it failed" before the agent repeats it. The authors coin Memory-as-Governance for this Definition PROJECTMEM '26:

memory that does not merely answer the agent but acts on its next action. — PROJECTMEM PROJECTMEM '26, Abstract

It explicitly contrasts itself with passive codified context — "Codified Context prevents repetition passively… not from the project's logged failures and not per action" — positioning the gate as a more active intervention than a read-only instruction file Vasilopoulos '26. (Evidence is a 207-event, 10-project, two-month self-study with no causal claim — a mechanism demonstration, not an effect size.)

Persistence as a generator of context for the next agent

MemDocAgent '26 applies persistent memory to producing the context artifact itself. It reframes repository documentation as a long-horizon agentic task: a single agent traverses the whole repo in one trajectory with a shared, typed, key-indexed store (RepoMemory), dependency-aware traversal, and NLI-based cross-document conflict verification Proposal. Versus memoryless per-component pipelines that re-fetch the same files (~50% overlap) and contradict themselves (13% inconsistency), it reduces inconsistency 75.5% and read-time 41% (23.44s → 13.76s) Empirical. Its information sufficiency metric — can the docs alone regenerate the code? — even beats human-written docstrings, framing documentation as "the contextual grounding that both human developers and coding agents rely on." Persistent memory here is the means by which a high-quality, consistent codified-context artifact gets produced.

Knowledge as versioned, governed assets

SkillWiki '26 scales persistence to an ecosystem, treating agent skills as long-lived "capability assets" rather than ad-hoc prompts Definition:

While knowledge is managed through Wikipedia and software through GitHub, agent skills still lack an infrastructure for large-scale production, governance, and evolution. — SkillWiki SkillWiki '26, Abstract

A Skill Provenance Graph links knowledge sources → skills → executions → validations → versions; an 8-state lifecycle plus Git-style snapshots, diffs, reviews, and releases make evolution "traceable, reviewable, and reversible, preventing uncontrolled drift" Proposal. Maintenance is feedback-driven — "reflection memories" from repeated failures generate maintenance proposals — but always subordinate to humans: "User decisions always take precedence over autonomous governance actions." The one quantitative result is a production run converting 99 of 125 heterogeneous artifacts into governed skill candidates; there is no baseline SkillWiki '26 Empirical.

▦ The shared design choice: legibility over vector recall

All three reject opaque embedding stores in favor of structured, auditable persistence. PROJECTMEM is "no semantic/vector retrieval by design," choosing determinism and grep/diff-ability PROJECTMEM '26; SkillWiki keeps explicit knowledge↔skill provenance links and reversible Git-style history SkillWiki '26; MemDocAgent uses a typed, key-indexed store with O(1) hits and consistency verification MemDocAgent '26. Each prefers a recall mechanism a human can inspect over one that is merely similar.

⟡ My inference — not from the papers

Across this corpus, persistent memory for coding agents looks deliberately different from the vector-database memory common in the general agent-memory literature (this hub's separate LLM Agent Memory review, on the hub). My read of why: the Context Limits page showed coding agents are brittle exact-match machines, and the same logic applies to memory — a fuzzy semantic recall that returns an approximately right file path or a stale decision is worse than useless when a tool will match it exactly. So these papers converge on append-only logs, provenance graphs, and deterministic projections: persistence you can audit, diff, and trust. The natural endpoint, which PROJECTMEM names outright, is that memory stops being a store you query and becomes a gate that governs — which is the bridge to the next page. This is my synthesis; the evidence here (one self-study, one demo, one doc-gen study) is thin and I would not over-read it.

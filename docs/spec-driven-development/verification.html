03 · Verification & Gates

Acceptance over inspection

The corpus's most developed theme is what happens after an agent produces something. If generation is cheap and abundant, the load-bearing question is how output earns its way into the codebase. The recurring answer: replace plausibility and manual line-by-line review with structured, evidence-based acceptance — risk-tiered gates, deterministic enforcement of process, machine-checkable contracts, and security promoted to an explicit spec.

Conversation discovers intent; artifacts and evidence earn acceptance

Koch states the principle the cluster shares Theory Proposal:

Conversation is useful for discovering intent; structured artifacts are required for implementation; evidence is required for acceptance. — Agentic Agile-V Koch '26, §Introduction

Agentic Agile-V proposes a two-layer process: a macro Agile-V lifecycle keeping each increment traceable to requirements/design/verification/approval, and a micro SCOPE-V task loop — Specify, Constrain, Orchestrate, Prove, Evolve, Verify Definition. Two mechanisms do the work. A conversation-to-contract gate:

Do not let an agent implement from a long chat. Let it implement from a reviewed brief. — Koch '26, §When Structure Is Mandatory

And risk-adaptive acceptance gates R0–R3 (exploratory → high-assurance), each with required evidence and a defined human gate, backed by an "evidence bundle" (task brief + requirement IDs, agent plan + affected files, executed commands/test results, diff summary + residual risks, acceptance-criteria-to-tests trace, reviewer decision, rollback path) Proposal:

Agent output is not accepted because it is plausible; it is accepted because it satisfies evidence appropriate to its risk level. — Koch '26, §Risk-Adaptive Acceptance Gates

Koch names the cost of not doing this — verification debt Definition: the accumulation of weak tests, hidden regressions, broad patches and reviewer burden "when output volume grows faster than verification capacity" Koch '26. (Agentic Agile-V is a bounded evidence synthesis; its quantitative figures, e.g. an AIDev count of 932,791 agent-authored PRs, are cited from other work.)

TDD as runtime-enforced governance

Hasanli operationalises the same instinct at the level of the orchestration engine, turning TDD from advice into an enforced protocol Proposal. The architecture separates non-authoritative model output from a deterministic engine that alone mutates state:

Language models never directly write to the file system. Instead, they return structured patch proposals that are subject to validation gates prior to application. — TDD Governance for Multi-Agent Code Generation Hasanli+ '26, §Governance-Centric Architecture

Every proposal passes four gates before any mutation — structural/schema validation, policy enforcement (disallowed paths), phase-consistency checks, and optional human/rule-based approval — and a bounded repair loop caps GREEN-phase retries at N=3, terminating early when a "failure signature" (exception type + failing tests + normalized message) repeats Hasanli+ '26. The motivation is LLM non-determinism: the paper cites a result that identical prompts yield "zero equal test outputs in up to 75.76% of cases" and that temperature 0 does not guarantee determinism (a cited figure, not its own).

◇ This one is a design, not yet a result

Hasanli is explicit that evaluation is preliminary — "initial experimentation suggests that explicit phase separation and validation gating reduce unstable retry cycles compared to baseline prompting," with no controlled metrics of its own yet Hasanli+ '26. Read it as an architecture proposal awaiting empirical test.

Machine-checkable contracts: fuzzing as the oracle

Suresh's Spec-Agent provides the corpus's strongest evidence that the "acceptance by evidence" loop can be automated for formal specs. It synthesises pre/post-conditions (up to first-order separation logic) across million-line C++ repos, selecting the target logic by static+dynamic analysis, then validating each candidate by fuzzing Proposal Empirical:

we repurpose [fuzz testing] as a strong pseudo-oracle for specification inference: any input that violates a candidate contract constitutes direct evidence that the contract is wrong. — Agentic Separation Logic Specification Synthesis Suresh+ '26, §Introduction

The loop is deterministic (analysis selects the language; each attempt is one LLM call; the fuzzer runs after every attempt, refining on counterexamples). It produces valid specs for up to 85% of functions with no false positives observed under fuzzing and expert review, while outperforming Claude Code Opus 4.6 in validity at ~10× lower token cost Suresh+ '26. Notably, fuzzing rejects contracts that already passed existing unit tests (8.0% of survivors on one repo) — evidence that the gate adds signal beyond the tests a project already ships. Full numbers on the Evidence page.

Security as an explicit spec, not a model prior

Grynets applies "acceptance by evidence" to security, arguing that if functional behaviour is specified explicitly, security behaviour must be too Theory Empirical:

If functional correctness requires explicit specification, security behavior should not be left to model priors. — Specification-Driven Development Benchmark Grynets+ '26, §Introduction

The paper makes security a traceable spec artifact (a Multilayer Specification Security Model) and tests AI-generated FastAPI backends against a hidden 221-test suite under three conditions. Modal failures fell monotonically: baseline 50 → generic standard (ASVS) 42 → application-specific security model 36 (pass rates 77.4% → 81.0% → 83.7%), with the largest gains in application-specific categories (Business Logic 17→9, Admin Safety 2→0) Grynets+ '26. The deliberately limited claim: "generic security standards help, but application-specific security modeling is needed." Importantly, the best condition still failed 36/221 tests, so the model is "a security-requirement operationalization layer, not… a guarantee of secure code."

The provocation: retire human code review

Monperrus takes the throughput argument to its conclusion — a position paper, explicitly "not a new empirical study," arguing every stated goal of code review can be met by agents Proposal:

The crossover point, at which the marginal benefit of human review no longer justifies its marginal cost, has already been reached. — The End of Code Review Monperrus '26, §Claim 3

His core structural argument is that mandatory human review of agent-written code collapses into rubber-stamping ("the human approves because the code looks correct, because the tests pass"), and should be replaced by an agent-in-the-loop verification pipeline, with humans reserved for high-risk/novel/regulated changes and developers becoming "specifiers and orchestrators" Monperrus '26. He raises his own counter-arguments as open risks rather than rebutting them away:

⚠ The argument's own admitted holes

Monperrus flags hallucination / silent false-negative approvals; correlated blind spots when the same model family writes and reviews; and prompt injection as "a qualitatively new attack surface that does not exist for human reviewers" with "no fully solved defences" Monperrus '26. These are the same correlated-reviewer and adversarial risks that recur on the Limitations page.

⟡ My inference — the gate is the real unit of spec-driven development

Across Koch, Hasanli, Suresh, and Grynets, the spec only does work because something mechanically checks against it: a fuzzer, a hidden test suite, a deterministic engine, a risk-tiered evidence bundle. The spec-driven SDLC these papers describe is really a closed loop — spec → generation → automated check → counterexample/repair — where the human supervises the loop rather than inspecting each output. Monperrus is the limit case of that logic; Grynets is its cautionary counterweight (even an explicit security spec left 36/221 failures). My read: the corpus's center of gravity is not "write better specs" but "make acceptance automatic and evidence-bound." Chips mark what I reason from, not agreement.

Page 02 · memory as a learned policy

Learning to remember

A recurring move in this corpus is to stop hand-coding memory heuristics — "summarise every N turns," "retrieve top-k by cosine" — and instead train the memory behaviour: what to write, what to keep, what to retrieve, and which past experience to trust. The supervision signals differ wildly, and that turns out to be the interesting part.

⟡ The question this page asks

If memory should be learned, the binding question is where the training signal comes from. The papers below answer it five different ways — masked reconstruction, task reward, novelty, verified outcomes, and oracle gating — and comparing those answers is my organising lens, not a claim any one paper makes.

Self-supervised context memory — supervise the process, not just the outcome

MemTrain trains a model's fixed-size context memory on unlabeled Wikipedia with two coupled proxy tasks: end-to-end masked reconstruction (an outcome signal — recover a masked entity after reading chunked passages) and intermediate memory recall (a process signal — at a random mid-step, recover an earlier masked fact from the current memory state). MemTrain '26 Proposal The process signal matters: removing it drops average accuracy from 70.31% to 63.28%, and the framework generalizes across a 32× context increase (a 21-point drop for the baseline shrinks to ~8.6 points). MemTrain '26 Empirical Its sharpest claim is that this is pre-training for memory: it raises a ceiling that simply running more downstream post-training cannot reach. MemTrain '26 Empirical

End-to-end rewards alone are often coarse and may not sufficiently constrain the quality of intermediate memory states. — MemTrain MemTrain '26, §Intermediate Memory Recall

Learning what to memorize from task reward

TaskMem reframes memory generation as a reinforcement-learned policy rather than a fixed summarization step, in two phases: first train for general memory quality, then — after deployment — adapt online with a tiny 2,048-parameter steering vector so the policy learns what is worth remembering for this task. TaskMem '26 Proposal On streaming video QA it improves accuracy by 5.3–7.0 points over its backbone, and the adapter learns task-specific focus — only the matched task's adapter helps, others don't — while resisting the catastrophic forgetting that full fine-tuning induces. TaskMem '26 Empirical A "richness reward" is needed to stop the policy reward-hacking toward short, accurate-but-empty memories. TaskMem '26 Empirical

memory is not merely a passive storage system, but an active, goal-driven process. An effective agent should continuously retain information that maximizes future utility. — TaskMem TaskMem '26, §Introduction

Supervising latent memory with novelty

Latent memory is efficient but, as JAMEL notes, hard to supervise because its vectors have no human-readable semantics — and ordinary task demonstrations don't supply step-level signal. Its fix is a deterministic, annotation-free novelty reward: in the GUI domain, code coverage is a clean proxy for "have I seen this before," and a persistent coverage baseline (a state never registers as novel twice) turns into a self-generated curriculum. JAMEL '26 Proposal Trained this way, the latent memory matches closed-model exploration depth while spending a fraction of the tokens. JAMEL '26 Empirical

memory and exploration form a mutually dependent loop … novelty-seeking interaction provides the supervision needed to make memory useful for future exploration. — JAMEL JAMEL '26, §Abstract

Learning to tune the memory system itself

EvolveMem (also on Architectures) belongs here too: instead of learning what to store, it learns how to retrieve, using an LLM to diagnose per-question failures and rewrite the retrieval configuration. The control comparison is the telling one — swapping the LLM diagnosis for random search over the identical action space costs ~9.6 F1, so the learning signal is the model's reading of why a query failed, not just trial-and-error. EvolveMem '26 Empirical The authors frame the loop as an autonomous "observe–hypothesize–experiment–validate" cycle replacing manual tuning. EvolveMem '26 Proposal

Posterior belief over skills — learn only from verified outcomes

Bayesian-Agent treats reusable harness assets (skills, SOPs, failure patches) as Bayesian hypotheses about whether a frozen model will succeed, and updates belief only from externally verified trajectory outcomes — explicitly not from the model's own self-assessment. Bayesian-Agent '26 Proposal The posterior maps to five inspectable actions (explore, patch, split, compress, retire), and a deliberately negative case study shows repeated failures driving a skill toward "retire" rather than hiding the rot. Bayesian-Agent '26 Empirical Importantly, it documents that learned skill-memory growth is non-monotonic: full online evolution sometimes underperforms the static baseline when evidence is still sparse. Bayesian-Agent '26 Empirical

the LLM may propose explanations or repairs, but the belief state is updated by externally checked evidence. — Bayesian-Agent Bayesian-Agent '26, §Trajectory Evidence

Learning what is worth storing — per user

Personalize-then-Store attacks the write decision: which sessions are even worth committing to memory? Its claim is that "worth-storing" is user-specific, not domain-intrinsic — the same domain is long-horizon for one user and transient for another — so a universal policy misallocates a fixed budget. PerMem '26 Definition It introduces session-level storage gating and shows that perfect (oracle) personalized gating yields large retention gains, especially under tight budgets — but that current learned/heuristic gating cannot realise them, and often underperforms storing everything. PerMem '26 Empirical

The gap between Oracle and the best gating method thus represents unrealized potential — not an argument against personalization, but a precise measure of how much headroom remains. — Personalize-then-Store PerMem '26, §Finding 3

⟡ What the signals have in common — my reading

The five training signals line up on a single axis: how cheap and how trustworthy the supervision is. Masked reconstruction (MemTrain) and code-coverage novelty (JAMEL) are nearly free and fully automatic but indirect. Task reward (TaskMem) and verified outcomes (Bayesian-Agent) are more faithful but need an environment with a grader. Oracle gating (PerMem) is the most faithful and the least available — which is exactly why PerMem can only measure the headroom, not capture it. My takeaway: the papers that work end-to-end are the ones that found a deterministic, automatic proxy for "good memory" (reconstruction, coverage); the ones that stall are waiting on a signal that doesn't exist cheaply yet (per-user worth, real-world task reward).

◇ Open problem the corpus inherits

Learned memory shifts the bottleneck from engineering to reward design. Two papers here explicitly hit reward-hacking or non-monotonic learning (TaskMem's empty-memory hack, Bayesian-Agent's online regressions), and none demonstrates a single learned-memory recipe that transfers across domains without a domain-specific signal. That gap is developed on Limitations.

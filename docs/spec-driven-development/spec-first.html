01 · Spec-First

The specification as the primary artifact

The defining claim of this corpus is that the specification stops being documentation about the system and becomes the thing you actually author — the contract an agent implements, tests against, and is sandboxed by. This page covers what these papers mean by "specification," the landscape of frameworks that operationalise it, and the recurring requirement that intermediate spec artifacts be grounded in reality rather than trusted on sight.

"Spec-centered," in their own words

The strongest statement of the thesis is Marron's "spec-centered" development, in which the spec is the root from which everything else flows Proposal:

specifications are the primary artifact of the software development process, and from them, flows the implementation, testing, deployment, and interaction with AI agents, data, and other systems. — Specifications for Humans, Agents, and Tooling Marron '26, §Introduction

For Marron, that means an API specification should be "more akin to a programming language than simply a description of a data-transport schema" Marron '26. The BAPI ecosystem makes the spec carry data and behaviour and resource constraints: executable pre/post-conditions written in the Bosque language, validated-string types that cannot trigger ReDoS, an env={…}/permissions={…} sandbox using URIs and globs, and a first-class tamper-proof $events log for temporal constraints Definition Proposal. A motivating question frames why this matters once an agent is the consumer:

Would you trust your AI agent to use the API below to make payments from your bank account? — Marron '26, §Introduction

▦ Specs for agents must be multi-modal Proposal

Marron argues an effective spec for agentic development combines natural language + examples + logical constraints, and adds an example keyword so examples serve as both tests and agent context. The classic case: a sort spec that guarantees isSorted but forgets output-is-a-permutation — a single example communicates the intent the logic missed and stops an LLM generating a plausibly-wrong implementation Marron '26.

This intent-as-artifact framing recurs well beyond Marron. De La Cruz observes that "intent clarity becomes more scarce as code production becomes less scarce" De La Cruz '26; Alenezi describes prompts functioning "like executable specifications with clear constraints and guardrails" Alenezi '26; and Mandl recommends structuring requirements with controlled language and explicit, measurable acceptance criteria (citing EARS) rather than free text Mandl+ '26.

The framework landscape

The clearest map of operational SDD tooling is de Macedo's process taxonomy, which scores six high-traction frameworks on six dimensions (specification, context, roles, execution, validation, portability), each 0 (absent) / 1 (partial) / 2 (strong) Definition Proposal. The headline finding: no framework scores 2 on all six — there is a structural trade-off between process depth and portability, and "specification" is the saturated common denominator that therefore distinguishes the frameworks least.

For context, the most-adopted framework, GitHub Spec Kit, had 106,786 stars in the May 2026 snapshot.

de Macedo's deeper point is a convergence across all six frameworks: "the unit of interaction is no longer the isolated prompt; it becomes an autonomous session over the repository" — persistent artifacts, work contracts, traceability and human review replace loose chat de Macedo '26. The central value of these frameworks, the paper argues, "lies less in automating the writing of code and more in how they structure the cognitive and operational work around the agent."

⚠ Without grounding, frameworks risk "institutionalising vibe coding"

de Macedo warns that lacking a base of context, validation and traceability, "AI frameworks run the risk of merely institutionalizing 'vibe coding' with an extra layer of terminology," and flags supply-chain risk in installable command/skill kits and platform lock-in de Macedo '26.

Context-grounding: specs are not enough if they are wrong

A spec-first workflow only helps if the spec matches the repository it targets. Taghavi names the failure mode directly — context blindness Definition:

the agent's intermediate artifacts can be internally coherent while being incompatible with the repository as it exists — Spec Kit Agents Taghavi+ '26, §Introduction

Spec Kit Agents augments GitHub Spec Kit's staged Specify → Plan → Tasks → Implement workflow with two hook types Proposal: read-only discovery hooks that probe repository state (globbing, grep, git history) before each phase, and validation hooks that check each artifact afterward. The key move is shifting verification earlier:

Rather than concentrating verification after implementation, we validate intermediate artifacts (SPEC/PLAN/TASKS) before code generation, catching hallucinated APIs, invalid paths, and architectural mismatches early. — Taghavi+ '26, §Verification, context-grounding, and tool-based validation

Empirically the effect is modest-but-real: judged quality rose from 3.51 to 3.66 (composite of completeness/correctness/style/maintainability, 1–5) on the 90-minute workflow, statistically significant on the paired completed-task subset (Wilcoxon, p<0.05), while maintaining 99.7–100% repository test compatibility; on SWE-bench Lite, grounding lifted Pass@1 from 56.5% to 58.2% Empirical Taghavi+ '26. The authors are careful that "the primary benefit is not a dramatic jump in average score, but earlier detection and prevention of compounding context errors." See the Evidence page for the full ledger and caveats (including a blinded human-preference test that did not favour the grounded variant).

⟡ My inference — "spec-first" is really "grounded-spec-first"

Reading these together, the corpus's operative unit is not the spec alone but the spec plus a binding to repository reality and to executable checks. Taghavi grounds the artifact against the repo; Marron makes the spec itself executable and sandboxed; de Macedo finds the frameworks that survive are the ones that add context, validation and traceability around the prompt. A spec that an agent cannot mechanically check against the world is, on this corpus's logic, closer to a prompt than to a program. Chips here mark what I reason from, not papers that state this conclusion.

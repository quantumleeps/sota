State of the Art · LLM Agent Memory

Agent memory stops being a store and starts being a process

Across these 14 papers — all posted between April and June 2026 — the centre of gravity of LLM-agent memory has moved away from "write a fact, retrieve a fact." The recurring design treats memory as something that evolves, is increasingly learned rather than hand-built, and — above all — must be actively kept correct and used at the right moment. Storage is cheap; using the right memory at decision time is the hard part.

What recurs across these 14 papers

Six observations have enough independent support within this corpus to count as consensus here. Each links to the page that develops it.

1 · Memory designs are shifting from static stores to self-evolving structures Proposal

The default is no longer write-once / retrieve-later. Memory restructures itself: a heterogeneous graph whose connectivity is pruned and consolidated FluxMem '26, a retrieval pipeline that self-tunes via an LLM-driven diagnosis loop EvolveMem '26, an append-only patch history that tracks how memory changed EvoArena '26, and a layered store that consolidates episodes into reusable SOPs and code GenericAgent '26. → Architectures

2 · Memory is increasingly learned, not engineered Empirical

Rather than hand-coding write/retrieve heuristics, several papers train the memory behaviour itself: a self-supervised objective for context memory MemTrain '26, an RL policy for what to memorize TaskMem '26, a novelty-supervised latent memory JAMEL '26, and a Bayesian posterior over reusable skills updated from verified outcomes Bayesian-Agent '26. → Learning

3 · Storage is not the bottleneck — use-at-decision-time is Empirical

Writing and recalling more does not reliably help. A lifecycle benchmark finds high memory-storage quality does not translate to better answers WorldMemArena '26; another shows updated evidence is retrieved yet still fails to govern the response STALE '26; structured retrieval over organised memory beats dumping the full context MemDreamer '26. → Consistency

4 · Keeping memory current is a distinct, mostly-unsolved problem Empirical

Staleness and implicit conflict are pervasive failure modes STALE '26; agents collapse memory to a single latest state and break when versions diverge EvoArena '26; most systems are weak at updating and distractor rejection, accumulating rather than maintaining state WorldMemArena '26. Append-only accumulation is the shared anti-pattern. → Consistency

5 · Memory is governed by a token / cache budget Theory

Efficiency is a first-class design driver: cache-aligned eviction that protects KV-cache prefixes TokenPilot '26, maximizing decision-relevant information per token GenericAgent '26, a fixed-size learned memory MemTrain '26, and latent compression of full trajectories JAMEL '26. → Architectures · Evidence

6 · Evaluation is being rebuilt around dynamics and lifecycle Definition

New benchmarks measure memory as a process under change, not static recall: evolution chains EvoArena '26, a write→maintain→retrieve→use lifecycle WorldMemArena '26, implicit conflict STALE '26, and personalized retention over years PerMem '26. → Evidence

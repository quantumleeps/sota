The 14 papers

Every citation chip on this site resolves to an entry below, and each entry links onward to the paper on arXiv — the chain is always claim → chip → bibliography → arXiv. This is a review of these 14 papers and nothing else: all are 2026 arXiv preprints (April–June 2026), gathered in June 2026 by an automated relevance search and curated to 14. Many are position or synthesis papers, not controlled studies; the role badges and the per-entry notes say which.

⟡ How this corpus was assembled — read this first

These papers were surfaced by an automated arXiv/OpenAlex relevance search for "spec-driven development" and "AI software development lifecycle", filtered to a 1 January 2026 date floor (all 14 land April–June 2026), ranked, and curated to 14 by topical fit. It is a search-shaped sample of recent, mostly un-peer-reviewed preprints — not a census of the field. Several entries are vision/position papers; weight them accordingly.

Spec-as-artifact & frameworks

CoreSpec Kit Agents: Context-Grounded Agentic Workflows
arXiv:2604.05278 · Pardis Taghavi, Santosh Bhavani, 2026 · system/tool (empirical) · cite: Taghavi+ '26
Augments GitHub Spec Kit's staged Specify→Plan→Tasks→Implement workflow with a multi-agent orchestrator and pre/post-phase "context-grounding" hooks that validate intermediate SDD artifacts against repository reality. Names the failure mode of "context blindness" and tests it across 128 runs and SWE-bench Lite.
arXiv abstract ↗
CoreSpecifications for Humans, Agents, and Tooling
arXiv:2606.15084 · Mark Marron, 2026 · system/tool + position · cite: Marron '26
A "spec-centered" manifesto plus the Bosque API (BAPI) ecosystem: treat the specification as the primary artifact from which implementation, tests, deployment and agent interaction flow. Adds executable pre/post-conditions, URI/glob permission sandboxing, a tamper-proof event log for temporal constraints, and multi-modal (NL + examples + logic) specs aimed at trustworthy agentic execution.
arXiv abstract ↗
CoreFrom Prompt to Process: a Process Taxonomy and Comparative Assessment of Frameworks Supporting AI Software Development Agents
arXiv:2606.04967 · Sanderson Oliveira de Macedo, 2026 · survey / framework · cite: de Macedo '26
A six-dimension process taxonomy (specification, context, roles, execution, validation, portability) and a 0–2 rubric applied to six high-traction SDD frameworks (GitHub Spec Kit, OpenSpec, BMAD, GSD, Spec Kitty, Reversa). Finds no framework strong on all six dimensions and that adoption (stars) and process completeness are orthogonal.

The agentic lifecycle

CoreAgentic AI in the Software Development Lifecycle: Architecture, Empirical Evidence, and the Reshaping of Software Engineering
arXiv:2604.26275 · Happy Bhati (Northeastern University), 2026 · survey · cite: Bhati '26
A synthesis that reframes the central object of inquiry from "code generation" to "delegated execution under human supervision," proposes a six-layer reference architecture (L0–L5) and an Agentic SDLC (A-SDLC) mapping each classical phase to agent equivalents, and compiles SWE-bench and productivity evidence from industry/academic reports. Conceptual contribution; no original measurement.
arXiv abstract ↗
CoreAI-driven Software Development: A Pragmatic Path to Agentic Development Processes
arXiv:2606.15283 · Peter Mandl (HM Munich), Paul Mandl (Findustrial), 2026 · framework / position · cite: Mandl+ '26
A pragmatic transformation framework (AI-assisted → AI-integrated → AI-driven/agentic) organised around a "harness" — the technical and organisational embedding (context, tool access, verification, permissions, logging, human approval) that turns model capability into governed workflow. Grounded in one exploratory case study; recommends EARS-structured requirements and DORA-plus metrics.
arXiv abstract ↗
CoreBridging Requirements and Architecture: Multi-Agent Orchestration with External Knowledge and Hierarchical Memory (MAAD)
arXiv:2606.01385 · Ruiyin Li, Xiyu Zhou, Yangxiao Cai, Peng Liang, Jifeng Xuan, Zhi Jin (Wuhan U.); Yiran Zhang, Weisong Sun, Yang Liu (NTU), 2026 · system/tool (empirical) · cite: Li+ '26
MAAD orchestrates four role-specialised agents (Analyst, Modeler, Designer, Evaluator) with RAG and a three-layer memory to transform a Software Requirements Specification into multi-view "4+1" architecture blueprints with quality-attribute (ATAM) assessment — pushing spec-driven automation past code into the architecture phase. Evaluated on 10 SRS cases vs MetaGPT and with 6 practitioners.

Verification, gates & governance

CoreAgentic Agile-V: From Vibe Coding to Verified Engineering in Software and Hardware Development
arXiv:2605.20456 · Christopher Koch (independent), 2026 · position / framework · cite: Koch '26
Argues the bottleneck has moved from prompting to process control, and proposes a two-layer framework: an Agile-V macro lifecycle plus a SCOPE-V task loop (Specify, Constrain, Orchestrate, Prove, Evolve, Verify), with a conversation-to-contract gate, a minimum-input-artifact model, and risk-adaptive acceptance gates (R0–R3) backed by "evidence bundles." A bounded evidence synthesis, not a new study.
arXiv abstract ↗
CoreTDD Governance for Multi-Agent Code Generation via Prompt Engineering
arXiv:2604.26615 · Tarlan Hasanli, Shahbaz Siddeeq, Bishwash Khanal, Pyry Kotilainen, Tommi Mikkonen, Pekka Abrahamsson (Jyväskylä / Tampere), 2026 · system/framework (preliminary) · cite: Hasanli+ '26
Encodes classical TDD (Red-Green-Refactor) as runtime-enforced governance for multi-agent code generation: a machine-readable "manifesto" of TDD principles plus an architecture that separates non-authoritative LLM proposals from a deterministic engine holding sole authority over state mutation, phase ordering, validation gates, and bounded repair. Evaluation is explicitly preliminary. EASE 2026.
arXiv abstract ↗
CoreSpecification-Driven Development Benchmark: Security Knowledge Transition
arXiv:2606.00167 · Oleg Grynets, Andrii Salyk, Vasyl Lyashkevych, Oleh Kaskun, Danyil Zhuravchak (EPAM Systems), 2026 · benchmark (empirical) · cite: Grynets+ '26
Makes security an explicit, traceable specification artifact for AI code generation: a Multilayer Specification Security Model and a Security Knowledge Transition Method, with a hidden-oracle benchmark. Empirically, generic standards (ASVS) help and application-specific security modelling helps further — observable security failures fell baseline → ASVS → Multilayer on a hidden test suite.
arXiv abstract ↗
CoreAgentic Separation Logic Specification Synthesis (Spec-Agent)
arXiv:2605.27531 · Tarun Suresh, David Korczynski, Julien Vanegue (Bloomberg), 2026 · system/tool (empirical) · cite: Suresh+ '26
Synthesises formal code contracts (pre/post-conditions, up to first-order separation logic) across million-line C++ codebases, using an adaptive spec-language selector and fuzz-testing as a pseudo-oracle with counterexample-guided refinement. Reports valid specs for up to 85% of functions at ~10× lower token cost than a frontier coding agent — the corpus's formal-methods leg of "spec-driven."

Evaluation

CoreSpecBench: Evaluating Specification-Level Reasoning for Software Engineering LLM Agents
arXiv:2605.30314 · Grant Hamblin, Kevin Song, Zhanda Zhu, Anand Jayarajan, Sihang Liu, Nandita Vijaykumar, Gennady Pekhimenko (U. Toronto / Waterloo / Vector / NVIDIA), 2026 · benchmark · cite: Hamblin+ '26
Argues existing SWE benchmarks assume the spec is correct and test only implementation; SpecBench instead measures "specification-level reasoning" — finding deficiencies in initial RFC design proposals before code, scored against expert maintainers' historical critiques across five ecosystems (Kubernetes, React, Rust, TVM, vLLM). Best agent reaches 44.4%.

The intent-centric turn & the human role

CoreFrom Code-Centric to Intent-Centric Software Engineering: A Reflexive Thematic Analysis of Generative AI, Agentic Systems, and Engineering Accountability
arXiv:2605.11027 · Elyson De La Cruz (U. of the Cumberlands), 2026 · empirical (qualitative discourse analysis) · cite: De La Cruz '26
A reflexive thematic analysis of public + scholarly discourse, deriving eight themes and a five-stage maturity pathway around the shift to "intent-centric" engineering — humans specify intent, constraints, and acceptance evidence while agents generate/test/repair. Names "bounded autonomy" and "governance and provenance debt"; central tension is acceleration vs accountability.
arXiv abstract ↗
SupportHuman-AI Collaboration and the Transformation of Software Engineering Work
arXiv:2606.03394 · Mamdouh Alenezi (SDAIA), 2026 · framework / interpretive synthesis · cite: Alenezi '26
A theory-building synthesis arguing the engineer's durable value moves from code volume to intent specification, critical judgment, and accountable oversight. Contributes a Future Software Engineering Competency Framework (FSE-CF) and three coexisting paradigms (Traditional / GenAI-enabled / Agentic), with intent specification ranked the least AI-substitutable, highest-criticality competency.
arXiv abstract ↗
SupportThe End of Code Review: Coding Agents Supersede Human Inspection
arXiv:2606.13175 · Martin Monperrus (KTH), 2026 · position / vision · cite: Monperrus '26
A deliberately provocative position paper arguing every stated goal of human code review can be met by coding agents at lower cost and higher throughput, so mandatory human review should be replaced by an agent-in-the-loop verification pipeline — and developers become "specifiers and orchestrators," relocating scrutiny upstream to requirements and downstream to monitoring. Synthesis, not a new study.

01 · Instruction Files & Codified Context

The repository instruction file, and what scales past it

The AGENTS.md/CLAUDE.md file is the canonical artifact of context engineering: durable, version-controlled, machine-targeted repository knowledge an agent reads before acting. Three papers in this corpus interrogate it directly — one builds a tiered infrastructure around it, two stress-test it — and they do not entirely agree. That disagreement is the most useful thing on this page.

The artifact: documents written for a machine reader

In Vasilopoulos '26, codified context is defined as "structured artifacts written explicitly for machine consumption — documents whose primary audience is an AI agent, not a developer" Definition. Configuration files like AGENTS.md and CLAUDE.md are described elsewhere in the corpus as having "emerge[d] as a standard for context injection and for providing a persistent memory for coding agents" dos Santos+ '26. But how that knowledge is delivered matters: a source-level reading of Claude Code finds CLAUDE.md is injected as a user-context message, not a system prompt, so "model compliance with these instructions is probabilistic rather than guaranteed" — a deliberate split between guidance (the file, probabilistic) and enforcement (permission rules, deterministic) Liu+ '26 Definition.

Scaling past a single file: codified context as infrastructure

Vasilopoulos argues the single manifest does not survive a large codebase, and reports building a three-tier codified context infrastructure for a 108,256-line C# project written with Claude Code as the sole code generator over 283 sessions Empirical Vasilopoulos '26:

0.6%
Tier 1 — constitution: one always-loaded ~660-line file of conventions, retrieval hooks, and routing. Vasilopoulos '26
8.6%
Tier 2 — 19 specialist agents, ~9,300 lines; over half of each spec is project-domain knowledge, not behavior. Vasilopoulos '26
15.0%
Tier 3 — knowledge base, 34 docs ~16,250 lines, retrieved on demand (1,478 MCP retrievals across 218 sessions). Vasilopoulos '26

In total, machine-targeted documentation reached 24.2% of the codebase (~26,200 lines across 54 files), maintained at roughly 1–2 hours/week, with meta-infrastructure prompts (building the architecture itself) accounting for 4.3% of substantive prompts Empirical Vasilopoulos '26. The work is a single-developer, single-project experience report that explicitly disclaims any causal effect — useful as a mechanism narrative, not as evidence of impact.

⚠ Staleness is named the primary failure mode

The same report flags the central danger of codified context: "agents trust documentation absolutely and out-of-date specs cause silent failures." On at least two occasions, stale specs caused the agent to generate code conflicting with recent refactors (e.g. wiring a value through a deprecated path) Vasilopoulos '26 Empirical. Codified context is a liability the moment it diverges from the code it describes.

The skeptic's evidence: more context files can make agents worse

The strongest counterweight in the corpus is a controlled evaluation across four agent/model pairs on SWE-bench Lite and a purpose-built 138-instance benchmark (AGENTbench) of Python repos that ship developer-written context files Gloaguen+ '26:

…we find that AGENTS.md tend to reduce task success rates compared to providing no repository context, while also increasing inference cost by over 20%. — Evaluating AGENTS.md Gloaguen+ '26, Abstract
  • Auto-generated files are net-negative: LLM-generated context files reduced average resolution by 0.5% (SWE-bench Lite) and 2% (AGENTbench), causing drops in 5 of 8 model×benchmark settings, while adding 20–23% inference cost Empirical.
  • Developer files barely help: human-written files improved success ~4% on average vs −3% for LLM files — a marginal gain, not the headline win vendors imply Empirical.
  • It is not a compliance failure: instructions are followed — a tool named in the file (uv) was used 1.6×/instance vs <0.01× when unmentioned; repo-specific tooling 2.5× vs <0.05× Empirical. The files are obeyed; they just don't help.
  • They are largely redundant docs: only when all other documentation is removed do generated files help (+2.7%) — explaining why they seem useful in sparsely-documented repos Empirical.

The authors' recommendation is blunt: "human-written context files should describe only minimal requirements", and auto-generated ones should be omitted "for the time being" Gloaguen+ '26 Empirical. (Caveat from their own limitations: Python-only, single sample per agent, no variance reported.)

Hygiene: what is actually wrong with these files in the wild

If quality matters more than presence, what does poor quality look like? Mining the 100 most-starred repos with an AGENTS.md/CLAUDE.md file, dos Santos+ '26 derive six configuration smells and find at least one in 91 of 100 files (207 instances total; only 9 files clean) Empirical:

SmellWhat it isPrevalence
Lint LeakageRules already enforced by a linter/formatter, wasting context62%
Context BloatFile overloaded with rules/examples (flagged ≥200 lines; largest 1,477)42%
Skill LeakageRarely-used specialized instructions kept in the always-loaded file instead of on-demand skills35%
Init FossilizationGenerated by /init and never reviewed/updated (24 single-commit files)—
Blind ReferencesPointers to external files without explaining their purpose—
Conflicting InstructionsInternally contradictory directives (lowest detection precision, 57%)—

The smells co-occur: Conflicting Instructions together with Skill Leakage raise the likelihood of Context Bloat to 83% (association-rule confidence) Empirical. Init Fossilization is the provenance smell — the file is a frozen /init artifact carrying stale instructions, the same staleness danger Vasilopoulos names, now measured across real repos.

⟡ My inference — reconciling the tension, not from any one paper

This corpus contains a real disagreement: one paper builds context until it is 24.2% of the codebase Vasilopoulos '26, another shows that adding context files reduces success Gloaguen+ '26. My reading is that they are measuring different variables, and the reconciliation is consistent with the third: the harmful files are auto-generated, always-loaded, redundant, and unmaintained — exactly the smell profile dos Santos+ measure dos Santos+ '26. Vasilopoulos's volume sits mostly in Tier 3, loaded on demand, and is human-curated and actively maintained. So the corpus is not really split on "does codified context help" — it is split on presence vs curation, and curation (minimal always-loaded spine, on-demand depth, kept in sync with the code) is where the signal points. This is my synthesis; chips mark what I reason from, not papers that assert it.

02 · Finding & Provisioning Context

Getting the right code in front of the agent

Before a coding agent can edit, it has to find the code that matters. Three papers in this corpus treat that search as a first-class, measurable, separable problem — and converge on the same answer: pull exploration out of the solving loop and hand the solver a compact, structured artifact instead of a trail of grep output.

Exploration is the dominant, separable bottleneck

Repository search is not a rounding error in an agent's trajectory — it is the bulk of it. FastContext '26 measures, over 300 GPT-5.4 trajectories on SWE-bench Multilingual, that reading and searching account for 46.5% of the main agent's tokens and 56.2% of its tool-use turns, and that editing only begins at turn 8.47 on average — a median of six sequential exploration turns and 15.5 tool calls precede the first edit Empirical. The cost is not just tokens; it is that the solver carries all that navigation noise in its window.

That localization is also shallow. RepoMirage '26 finds that among GPT-5's resolved SWE-bench cases, 53.8% inspect only one file and 88.0% no more than three Empirical — strong scores often come from narrow, localized cues rather than genuine cross-file reasoning:

successfully resolving an instance in the original benchmark does not imply the ability to understand and reason across the context in the repository. — RepoMirage RepoMirage '26, Introduction

Context quality is measurable — and it predicts success

SWE-Explore '26 isolates exploration as a standalone, ranked, line-level context-selection task (848 instances, 10 languages, 203 repos), with ground truth derived from the intersection of ≥2 solution-verified agent trajectories Definition. Its headline results sharpen exactly where the lever is:

r = 0.950
Pearson correlation between an explorer's context efficiency and the downstream task resolve rate — context selection is causally tied to solving. SWE-Explore '26
0.14–0.19
Line-level recall of strong general agents, despite file-level HitFile of 0.64–0.68 — agents find the right files but miss the decisive lines. SWE-Explore '26
7–9 pp
Drop in resolve rate from random non-core context when evidence is scarce — yet missing core context hurts more than redundant context. SWE-Explore '26

Two consequences for context engineering. First, classical retrieval (BM25/TF-IDF) sits near random while agentic explorers form a clear tier above it — but the open frontier is line-level recall, not file finding Empirical. Second, the precision-vs-recall trade tilts toward recall once coverage is decent:

missing core evidence hurts more than moderate redundant context. — SWE-Explore SWE-Explore '26, Conclusion

A notable secondary finding: five general agents with very different harnesses (Claude Code, Codex, OpenHands, Mini-SWE-Agent, AweAgent) converge to nearly the same exploration operating point — suggesting exploration capability is more model/mechanism-bound than harness-bound Empirical SWE-Explore '26.

"Exploration drift": access is not understanding

Why doesn't more searching help? RepoMirage '26 applies semantics-preserving perturbations (rerouting imports through proxy chains, masking runtime targets, externalizing constants) and watches agents fall apart: average resolved rate drops from 66.8% to 49.8% under perturbation, and to 25.3% on explicitly perturbation-derived tasks Empirical. Crucially, agents respond by accessing more files — 4.77 → 13.24 on average — without solving more, a failure the paper names exploration drift Definition:

…agents access broader repository context but fail to turn it into effective structure information. — RepoMirage RepoMirage '26, Abstract

The convergent fix: decouple exploration, emit a structured artifact

All three papers land on the same architecture — separate the search from the solve, and pass forward a compact, machine-consumable context object rather than raw history.

▦ Two instantiations of one pattern

A trained explorer subagent. FastContext delegates exploration to a dedicated 4B–30B model with three read-only tools that issues parallel searches and returns only a <final_answer> block of file paths and line ranges; the explorer's noisy observations never reach the solver. This improves end-to-end resolution up to +5.5% (GPT-5.4 SWE-bench Pro 46.0→51.5) while cutting main-agent tokens up to 60%, for ~2.1% overhead — and a compact 4B-RL explorer can beat a 30B-SFT one FastContext '26 Empirical.

A structure-first writing phase. RepoMirage's RepoAnchor runs an exploration stage that writes an intermediate INSTRUCTION.md (files inspected, their roles, key relations) before a separate solving stage consumes it. Supplying such compact structural hints lifts GPT-5 from 27.6→62.6 on the hardest tasks RepoMirage '26 Empirical — the gain is from provisioning the right structure, not merely granting access.

By moving broad exploration into a separate subagent, the main agent can receive cleaner repository evidence rather than carrying the irrelevant code accumulated during navigation. — FastContext FastContext '26, Introduction

⟡ My inference — not from the papers

The agent-authored INSTRUCTION.md in RepoAnchor and the <final_answer> evidence block in FastContext are the same idea as the hand-written instruction files on the previous page — codified, machine-targeted repository context — except generated just-in-time and scoped to one task rather than authored once and loaded always. My reading of the corpus is that "context engineering" is splitting into two registers: a durable, human-curated spine (the repo file) and a transient, machine-generated context object minted per task. The provisioning papers are quietly arguing the transient register is where the measurable wins are. One caveat I'd flag: FastContext and SWE-Explore share authors and infrastructure (overlapping SJTU/Microsoft groups), so their agreement is less independent than it looks — see Limitations.

03 · Learning to Orchestrate

Training the coordinator with reinforcement learning

If coordination is a policy, it can be learned. Seven papers in this corpus apply reinforcement learning to multi-agent orchestration, and they pull toward one organising idea: supervise the orchestration decisions — when to spawn, whom to delegate to, how to aggregate, when to stop — rather than only the final answer. The recurring obstacle is the same one classical multi-agent RL has always faced: credit assignment, now over a tree of LLM calls that is expensive to roll out.

The organising abstraction: the orchestration trace

▦ Reframe RL-for-MAS around a temporal event graph Definition

The corpus's spine survey replaces the per-agent trajectory with the orchestration trace — a temporal graph of spawn / delegate / message / tool / return / aggregate / stop events drawn from a "dynamic Dec-POMDP" that adds spawn/despawn dynamics. It organises the design space into eight reward families (R1–R8, with R7 "orchestration reward" — parallelism, finish-rate, split correctness — the one with no single-agent analogue), an eight-level credit-bearing-unit hierarchy (team → orchestrator → role → agent → turn → message → tool → token), and five orchestration sub-decisions O1–O5 C. Zhang '26.

RL must optimize not only individual actions, but also how work is spawned, delegated, communicated, aggregated, and stopped. — RL through Orchestration Traces C. Zhang '26, Abstract

▦ Two gaps in the pooled literature, as of May 2026 Empirical

From 84 retained entries the survey reports that no method explicitly trains the stopping decision (O5) as an RL target, and that explicit counterfactual message-level credit is near-absent (one entry). It also argues "the gap is not primarily algorithmic — it is in what shape of rollout and harness academic methods train against," noting multi-agent rollouts are "10–100× more expensive than single-agent" C. Zhang '26. These are survey-scoped counts over a curated pool, not field-wide prevalence.

Where the reward signal attaches

The empirical RL papers can be read as different answers to "what do we reward, and at what granularity?"

▦ Reward the plan before sub-agents run Empirical

Orch-RM trains a self-supervised, orchestration-level Bradley-Terry reward model that scores an orchestration plan — roles, role descriptions, collaboration structure, interaction graph — before any sub-agent executes, using preference pairs mined from intermediate training checkpoints. It improves training-token efficiency up to 10× while raising multi-agent test-time-scaling accuracy up to 8%, and beats the strongest trajectory-level LLM judge on BrowseComp+ at roughly 17× lower verification cost Tsang+ '26. It motivates this by noting MAS-Orchestra's GRPO "consum[es] over 1B tokens for only 100 update steps."

▦ Reward the step-level orchestration decisions Proposal Empirical

Orchestra-o1's DA-GRPO scores each main-agent decision (delegation, sub-agent/backend selection, tool use, answer) against expert trajectories with a four-dimension rubric reward, "instead of GRPO's final-answer-only reward" — using the expert decision "as a reference rather than the only correct answer." On Qwen3-8B it beats SFT and vanilla GRPO (30.0% vs 28.6% / 27.7% on OmniGAIA) F. Zhang+ '26.

▦ Reward only the outcome — and let routing emerge Empirical

Maestro takes the opposite tack: a GRPO policy trained with an outcome reward plus a format reward, "no step-level routing supervision," learns the compositional Model@@Skill routing on its own. RL contributes the bulk of the result (+17.4% over the untrained workflow), and removing the format reward costs −13.1% because malformed actions break the multi-turn protocol Wu+ '26. AgensFlow's online UCB1 is a non-RL point on the same spectrum — bandit acquisition over a folded task signature, with reward-signal auditability treated as part of the design Koenigstein '26.

▦ Collapse the roles into one model Proposal Empirical

Role-Agent uses a single LLM as both agent and environment: a World-In-Agent module predicts future states to form a multiplicative predictive reward (so it "cannot independently introduce extra credit" for failed trajectories), and an Agent-In-World module mines structured failure modes to reshape the task distribution — averaging over 4% gain with ~5.2% extra compute and no separate environment model Wang+ '26. It is dual-role, not multi-agent, but its credit-assignment and curriculum mechanics are squarely orchestration-relevant.

⟡ My inference — the field is reinventing credit assignment for LLM teams

Reading these together, the open problem the survey names theoretically — credit diffuses to noise under a shared terminal reward as the trace grows C. Zhang '26 — is exactly what the empirical papers route around in practice: Orch-RM moves the signal up to the plan, DA-GRPO and Role-Agent move it down to per-decision/per-step, Maestro keeps it terminal but adds a structural format reward. None of them, in this corpus, learns when to stop as an RL target, matching the survey's stated gap. I read "where does the reward attach" as the live design axis of LLM-MAS RL; chips mark the methods I reason from, not a consensus they declare.

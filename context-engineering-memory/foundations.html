From prompt engineering to a managed context

In 2025 the field gave itself a name. "Context engineering" reframes the prompt as just one part of a dynamically curated context, and "agent memory" reframes the store behind it as a persistent cognitive state. This page sets the definitions and shows where the two literatures meet — at the context window itself.

What "context engineering" came to mean

The foundational survey defines context engineering as a discipline that "transcends simple prompt design to encompass the systematic optimization of information payloads for LLMs" Mei+ '25 Definition. It reconceives the context not as a static string but as a dynamically assembled set of components — system instructions, knowledge, tools, memory, state, and the query — optimized under a hard length budget.

Context Engineering re-conceptualizes the context C as a dynamically structured set of informational components, c₁, c₂, …, cₙ … orchestrated by a high-level assembly function, A. — A Survey of Context Engineering Mei+ '25, §3.1

Industry arrived at the same framing from the practitioner side. Anthropic defines it as "the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference" Anthropic '25 Definition, and casts the goal as a search for economy:

good context engineering means finding the smallest possible set of high-signal tokens that maximize the likelihood of some desired outcome. — Effective Context Engineering Anthropic '25, "The anatomy of effective context"

Why the window is scarce: context rot and the attention budget

The premise underneath everything is that adding tokens is not free. Anthropic popularized two terms for this. Context rot: "as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases" Anthropic '25 Anthropic '26 (Cookbook) Definition. And a finite attention budget — every new token depletes it, a scarcity the post traces to the transformer's n² pairwise relationships over n tokens Theory.

The academic literature supplies the same constraints under other names: the "lost-in-the-middle" positional bias, and the O(n²) attention wall — the survey notes that scaling one 7B model from 4K to 128K tokens implies a "122-fold computational increase" (a figure it attributes to prior work, not its own measurement) Mei+ '25 Theory.

▦ The shared premise

Across practitioner and academic sources, the context window is treated as a finite resource with diminishing marginal returns — "a performance gradient rather than a hard cliff" Anthropic '25. This is the assumption that makes everything downstream — compaction, retrieval, memory hierarchies — necessary rather than optional.

The other half: agent memory as persistent state

If context engineering manages the window, agent memory manages what persists behind it. The memory survey defines it as the thing that "turns a stateless text generator into a genuinely adaptive agent," formalized as a write–manage–read loop and, more strongly, as a POMDP belief state — "a sufficient statistic of the interaction history for good action selection" Du '26 Theory. Its provocative claim:

the gap between "has memory" and "does not have memory" is often larger than the gap between different LLM backbones. — Memory for Autonomous LLM Agents Du '26, "Memory as a differentiator"

Two literatures, one boundary

The clearest statement of how the halves relate comes from the memory survey, which draws the line precisely Hu+ '25 Definition:

context engineering operates at the level of resource allocation and interface correctness … agent memory governs what the agent knows, what it has experienced, and how these elements evolve. — Memory in the Age of AI Agents Hu+ '25, §2.3.3

The overlap is working memory: the live management of the context window during a long interaction. That is exactly where this review concentrates — the techniques on the Practice and Memory Hierarchy pages are working-memory mechanisms; the stores on the Memory Systems page are the persistent layer behind them.

⟡ My inference — not from the papers

Read together, the corpus marks a convergence: by 2026 "context engineering" and "agent memory" are no longer separate conversations but two views of one system — the management of information across a tiered store whose fastest, scarcest level is the context window. The rest of this site treats them that way. No single paper makes this unification claim; it is the throughline I read across the three surveys and the systems papers, and I develop it as a reference model on the Synthesis page.

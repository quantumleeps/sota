06 · Security

The DevSecOps surface

Codified context is, by design, attacker-reachable input that the agent "trusts absolutely." That makes the context layer a security surface, not just a productivity one. The corpus treats this from three angles: a system-level threat model, the harness-level defenses one production agent actually ships, and the argument that trust — not capability — is the real gate on deployment.

Monitor the system, not just the model

The position paper here argues LLM-app security failures "should be treated as expected operational conditions rather than exceptional events," requiring an EDR-style, workflow-aware monitoring layer — because "the primary barrier to trustworthy deployment is not further improving model capability but establishing system-level threat monitoring" Zhang+ '26 Proposal. It models the reference system as an 8-stage MCP pipeline and calls MCP "the standardized architectural spine that connects AI agents to external data and tools" — which is precisely why it is also the place to instrument. Its key argument against the usual defenses:

guardrails fundamentally operate as localized control mechanisms… yet lack global visibility into cross-stage interactions and emergent failure modes that span the end-to-end execution workflow.— System-Level Threat Monitoring Zhang+ '26, §Alternative Views

This is a conceptual/position paper: the 14-threat taxonomy and monitoring artifacts are proposed and illustrated, not empirically validated. Zhang+ '26

Defense in the harness — and its gaps

Claude Code is the corpus's worked example of harness-level defense. Its central security property is the separation of reasoning from enforcement: the model only emits tool-use requests; the harness validates and dispatches, so "a compromised model cannot override sandboxing or deny-first rules" Liu+ '26 Theory. It layers a deny-first posture (deny > ask > allow), seven permission modes on a monotonic autonomy spectrum, and seven independent safety layers any one of which can block.

⚠ Documented gaps in that defense

The same analysis is candid that defense-in-depth can fail when layers share economic/performance failure modes: commands with >50 subcommands fall back to one generic permission prompt instead of per-subcommand deny checks (a documented bypass), and a pre-trust initialization window lets hooks/MCP/settings load before the trust dialog — outside the deny-first pipeline — implicated in several CVEs it cites (e.g. CVE-2025-59536, CVE-2026-21852) Liu+ '26 Empirical. The lesson: codified extensions (hooks, MCP, settings) are part of the trust boundary, and when they load matters as much as what they permit.

Trust as the adoption gate

The "programming with trust" opinion piece argues the binding constraint on deploying AI software engineers is developer trust, and that "the emphasis will be… increasingly on programming with trust" Roychoudhury+ '25 Theory. It splits the problem usefully:

Technical trust

correctness, security, performance, maintainability, standards compliance, testing, dependency management.

Human trust

explainability/transparency, ethical alignment, familiar patterns, expertise match, adaptation to feedback.

Its operational proposals map cleanly onto codified context as a trust mechanism: "transparent scaffolding" — tagging AI-generated code and embedding authorship provenance and confidence metrics in PR templates — and "review parity" — the same review gates and quality thresholds for agent code as for human code, backed by code-review agents with static/dynamic analysis Roychoudhury+ '25 Proposal. In other words, provenance and review are not just process hygiene; they are how an agent earns a "senior-colleague reputation."

The under-codified guardrail

⟡ My synthesis: the security gap is also a content gap

Put the file studies next to the threat model and a sharp problem appears. Context files are where teams could encode security expectations for agent-written code — yet across every study, security is the least-codified content category (8.7% of CLAUDE.md files '25a; 14.5% cross-tool '25b; 34% of Cursor repos Jiang & Nam '26; only 6 of 155 AGENTS.md sections Mohsenimofidi+ '26). So the artifact best placed to carry guardrails for agent code is precisely where guardrails are most often absent — while the threat surface those guardrails would address is expanding Zhang+ '26. For a DevSecOps reader this is the actionable gap: codify security/NFR expectations into the context layer and monitor the system-level signals, because neither the file nor the monitor suffices alone. This pairing is my recommendation, not a single paper's claim.

One rung up: agent identity

Finally, multi-agent ecosystems push trust into naming and discovery. NANDA notes that static endpoints "do not scale and are susceptible to DDoS attacks, overload, and flash mobs," and proposes resolving agent names to context-tailored endpoints with trust "relative to the reputation of the Name Space," enforced at registration Zinky+ '25 Proposal. It is speculative infrastructure, but it flags that in an A2A future, who an agent is talking to becomes a security decision the context layer must inform.

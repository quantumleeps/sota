Toward a context spine

⟡ Read this page differently

Everything here is my synthesis — a reading across the corpus, an extrapolation, and a set of opinions. No single paper makes these claims. Citation chips still appear, but here they mark what I'm reasoning from, not a source that agrees with my conclusion. The findings live on the other eight pages; this is the argument I'd build on top of them.

The claim

The corpus is, collectively, groping toward one object without quite naming it. The file studies measure a flat manifest; the anchor builds tiers; ACE makes it self-evolving; Confucius adds memory; the surveys give it a formalism; the security and failure papers describe how it breaks. Read together, they describe a single missing layer: a context spine — the unified, version-controlled, layered, retrievable, self-refreshing context substrate that sits between the codebase and the agent and is owned by the team.

"Codified context" today is mostly the top slice of that spine (a CLAUDE.md), written by hand and rarely maintained. The thesis I draw from the corpus is that the spine should be the unit of design — not any one file, agent, or retrieval trick — because every paper's contribution is really a different layer or property of it.

A reference model

This is my attempt to fold the corpus into one picture. Each layer cites the work I'm generalising from; the colour runs hot (always-loaded) to cold (retrieved on demand), with one cross-cutting governance plane.

Constitution
hot · always loaded
Conventions, build/test commands, architectural summaries, known failure modes, routing rules. The scaled CLAUDE.md. anchor Tier 1 file studies
Capabilities
explicit rules
Skills + MCP tools + scripts — the executable half of expert knowledge, loaded by need to protect the budget. explicit/tacit split skills vs MCP cost
Specialists

Sub-agent specs that embed domain knowledge and are auto-routed by trigger signals; isolated contexts returning summaries. anchor Tier 2 SOP roles
Knowledge base
cold · retrieved
Per-subsystem "knowledge about code" — design intent, constraints, failure tables — over a repo graph, fetched minimal-sufficient. anchor Tier 3 repo-RAG
Episodic memory
cross-session
Persistent notes incl. hindsight (failures/dead-ends), grown by delta and de-duplicated rather than rewritten. Confucius notes ACE deltas
Handoff contracts
between stages
Constraint-preserving payloads across planner→coder and agent→agent boundaries — the place most failures originate. planner-coder gap
Governance plane
cross-cutting
Versioning, ownership, drift detection, NFR/security codification, provenance, system-level monitoring, review parity. artifacts monitoring trust

This is a lens for organising the corpus, not a system anyone has built end-to-end. The anchor's three tiers are the closest real instance; the lower layers are assembled from other papers.

Eight bets I'd make

Where I think the evidence points, stated as design principles. Each is my opinion, grounded in the cited work.

1 · Treat the spine as config-as-code, not documentation

The strongest practical finding is that context files already "evolve like configuration code" and accrue "context debt" Chatlatanagulchai+ '25b. I'd make that explicit: version it, assign owners (CODEOWNERS), review it, and lint it in CI — checking that documented commands and conventions still hold. The "promptware engineering" and "Agentic SE artifact" papers give the discipline; the file studies give the urgency.

2 · Split explicit from tacit on purpose

The Siemens result — explicit procedural rules belong in executable tools, tacit principles belong in prose, and neither alone suffices Ulan uulu+ '26 — is the design rule that explains the file studies' biggest weakness: they over-encode "how" (commands, architecture) and under-encode "why" (rationale, constraints). A spine should route each kind of knowledge to its right layer rather than dumping everything into one markdown file.

3 · Grow-and-refine; never write-once

Staleness is the anchor's primary failure mode Vasilopoulos '26, context collapse is ACE's Zhang+ '26, and a quarter of real guidance files never get updated Robbes+ '26. My bet: every layer of the spine ships with a refresh mechanism — drift detection, reflector/de-dup, or hindsight capture — or it will rot, and an agent that "trusts documentation absolutely" will fail silently on it.

4 · Preserve constraints across handoffs, don't re-summarise

75.3% of multi-agent code failures are a planner→coder information loss Lyu+ '26. The most leveraged single intervention is therefore a structured handoff contract that carries core concepts, edge cases, and conditionals verbatim across stage boundaries — exactly what MetaGPT's typed artifacts did, and what loose natural-language summaries destroy.

5 · The budget is the constraint — minimise, de-dup, provenance-tag

The window is "the binding resource constraint" Liu+ '26; 28.7% of rule lines are wasteful duplicates Jiang & Nam '26; "good context is the minimum sufficient for the decision" Vishnyakova '26. A spine should retrieve minimal-sufficient context and tag each piece with its source — provenance is both a quality and a security control.

6 · Codify the guardrails that are currently missing

Security/NFRs are the least-codified content across every file study, while the threat surface grows Zhang+ '26. The cheapest security win available is to put NFR and security expectations into the context layer (templated), and pair them with system-level monitoring — because the file and the monitor each cover what the other misses.

7 · Invest in the harness; the durable value isn't the model

"Scaffold can beat model" Wong+ '25 and "98.4% harness" Liu+ '26 together say the engineering that compounds is in context management, tools, memory, and safety. Codified context is the declarative, ownable slice of that harness — the part a team can build and version without owning a runtime.

8 · Keep the human at the wheel — design for control, not autonomy

Professional developers succeed by controlling agents piecewise (2.1 steps/prompt) and reviewing everything Huang+ '25; trust, via provenance and review parity, is the adoption gate Roychoudhury+ '25. A spine should make plans, context, and provenance legible and editable by the human, not hide them behind autonomy.

Where I'd hold back

◇ Open bets I would not place yet

  • Schema standardisation. AGENTS.md is converging on a format, but "there is no established content structure yet" Mohsenimofidi+ '26. I'd keep the spine's schema loose and observed, not prescribed, until outcome-linked studies exist.
  • Self-evolving context at team scale. ACE is compelling but degrades without reliable feedback, and can be "polluted by spurious signals" Zhang+ '26. Auto-grown context shared across a team without a human gate is a risk I'd not take on yet.
  • More agents. MAST's 41–86.7% failure rates and "protocols are insufficient for inter-agent failures" Cemri+ '25, plus the survey's caution that multi-agent doesn't help when the prompt is robust Masterman+ '24, make me default to the fewest agents that work.
  • Quality, not just cost. The one controlled context-file study measured speed/tokens, not correctness Lulla+ '26. I treat "codified context improves code quality" as plausible-but-unproven.

The one-line version

The field has shown that context — codified, layered, maintained — is the dominant near-term lever on coding-agent behaviour. The unbuilt thing it keeps circling is a context spine: one owned, versioned, self-refreshing substrate that unifies the manifest, the tiers, the memory, and the governance into a single artifact a team can engineer.
